# MIMS - Make Idea Make Sense

> **アイデアを合理的、明確、実現可能に**

---

## プロジェクト概要

MIMSは「迷悟師」というAIエージェントであり、非技術ユーザーを対話によってソフトウェア設計に導き、曖昧なアイデアを明確なドメインモデルとインタラクティブなプロトタイプに変換します。

MIMSはClaude Code CLIおよびCodex Appユーザー向けに展開され、メインエージェントに注入されるペルソナ設定（`CLAUDE.md` / `AGENTS.md`）、構造化ワークフロー（Skill）、および原子タスクを処理するサブエージェントの3つの中核コンポーネントで構成されています。

### コアコンセプト

**Make Idea Make Sense** - 構造化された対話とAI支援を通じて、アイデアからプロトタイプへの変換を支援します：

- **アイデアを明確に** - 曖昧な要件を整理され、理解可能な概念に変換
- **アイデアを可視化** - 抽象的な概念を可視化・具体化
- **アイデアを実現可能に** - アイデアをインタラクティブなプロトタイプに変換

### 対象ユーザー

- **プロダクトマネージャー** - 製品アイデアを迅速に検証
- **ビジネスアナリスト** - ビジネス要件を明確に表現
- **起業家** - MVP（Minimum Viable Product）を計画
- **ドメインエキスパート** - 技術的背景なしでソフトウェアを設計

### 主な特徴

1. **技術的ハードルなし** - 日常言語で対話、ツールの学習不要
2. **AIによるガイダンス** - エージェントが積極的に質問し、要件を明確化
3. **自動モデル生成** - 対話からドメインオブジェクトモデルを自動構築
4. **即時可視化** - モデルの変化と関係をリアルタイム表示
5. **インタラクティブプロトタイプ** - ブラウザで直接実行可能なプロトタイプを生成

**MIMS** = **M**ake **I**dea **M**ake **S**ense

**発音**: /mɪmz/

**フルネーム**: Make Idea Make Sense

- **Make** - 作成、構築、...にする
- **Idea** - アイデア、概念、要件、インスピレーション
- **Make Sense** - 合理的、明確、整理された、理解可能になる

## 仕組み

```
┌─────────────────────────────────────────────────────────────┐
│                     MIMS 対話ループ                          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ユーザーがアイデア/要件を提示                                 │
│         ↓                                                   │
│  AIエージェントが質問（明確化、詳細化、補完）                    │
│         ↓                                                   │
│  ユーザーが回答                                               │
│         ↓                                                   │
│  エージェントがドメインモデルを自動更新                         │
│         ↓                                                   │
│  モデルをユーザーに表示して確認                                │
│         ↓                                                   │
│  ユーザーのフィードバック/確認                                 │
│         ↓                                                   │
│  モデルが明確で完全になるまでループ継続                         │
│         ↓                                                   │
│  インタラクティブプロトタイプを生成                            │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**モデリングフレームワーク**: MIMSは**FBS（Function-Behavior-Structure）**フレームワークを採用しています。これはデザインサイエンスにおける古典的な存在論（Gero, 1990）であり、UMLの3つのコア図（ユースケース図/状態図/クラス図）とも高度に対応しています。

| FBS層 | コアとなる問い | 対話ステージ |
|-------|--------------|-------------|
| **F 機能層** | 誰が使う？どんなシナリオ？何をする？ | 役割とシナリオ分析 |
| **B 振る舞い層** | どんな状態？どう操作？どんなルール？ | 状態と操作モデリング |
| **S 構造層** | 何を管理？どんな情報？どう関連？ | ビジネスオブジェクトと関係モデリング |

> 対話順序：トップダウン（F→B→S）、モデル出力：依存順序（S→B→F）

## 技術アーキテクチャ

```
対話レイヤー       データレイヤー       生成レイヤー       ドキュメントレイヤー
CLAUDE.md          YAMLモデル          HTML/JSプロトタイプ  Markdownドキュメント
+ Skill
+ サブエージェント
```

### 技術スタック

| 層 | 技術 | 理由 |
|---|-----|------|
| ペルソナと基本動作 | CLAUDE.md / AGENTS.md | T0でメインエージェントに注入、常に有効 |
| 対話ワークフロー | AI CLI Skill | オンデマンド読み込み、標準化トリガー |
| プロトタイプ/検証タスク | AI CLI サブエージェント | 原子タスク分離、コンテキスト汚染なし |
| データ保存 | YAML | 人間が読みやすく、AIに扱いやすい |
| プロトタイプ生成 | HTML/CSS/JS | 依存関係なし、ブラウザで直接実行 |
| ドキュメント出力 | Markdown | 読みやすく、バージョン管理が容易 |

## 3つのフェーズ

### フェーズ1：初期設計

対話を通じて完了（6ステップ）:
1. **P1 要件収集と準備** - 初期アイデアを理解、資料準備をガイド
2. **P2 資料理解**（任意）- ユーザー提供の資料を分析
3. **P3 コンテキストと目標** - ビジネス背景と全体目標を理解
4. **P4 ロールとシナリオ** - ユーザーロールとユースケースを定義（多対多）
5. **P5 ビジネスプロセス** - コアプロセスをマッピング（シナリオマウント、情報フロー）
6. **P6 アーキテクチャ概要** - モジュール分割、外部インターフェース、AI Agent評価

**チェックポイント**: 初期検証 → `srs.md` を生成

**出力**: `domain-model.yaml`（F層 + モジュール）+ `srs.md`（ソフトウェア要件仕様書）

### フェーズ2：詳細設計

詳細モデリング（5ステップ）:
1. **D1 ビジネスオブジェクト識別** - 管理すべきものとその属性を抽出
2. **D2 関係とモジュール割り当て** - 関係の定義、モジュール割り当て、AI Agent設計
3. **D3 状態とライフサイクル** - オブジェクトの状態と遷移条件を定義
4. **D4 操作とビジネスルール** - 操作とビジネスルールを定義
5. **D5 モデル検証と信頼度** - モデルの完全性を検証、信頼度を評価

**チェックポイント**: 完全検証 → `sdd.md` を生成

**出力**: `domain-model.yaml`（完全なFBS）+ `sdd.md`（ソフトウェア設計ドキュメント）

### フェーズ3：プロトタイプ生成

モデルから生成（9ステップ）:
1. **R1 モデル分析** - モデルデータの特徴を分析
2. **R2 ページ権限** - ロール別にページ権限を割り当て
3. **R3 ページ機能マッピング** - 操作をページ機能にマッピング
4. **R4 ページフロー設計** - ナビゲーションとページフローを設計
5. **R5 ページ構造** - データ特性に基づいてレイアウトを推奨
6. **R6 ページインタラクション** - インタラクションパターンを決定
7. **R7 コード生成** - HTML/CSS/JSを生成（プロトタイプサブエージェントに委託）
8. **R8 プロセス検証** - ビジネスプロセスでエンドツーエンド検証
9. **R9 納品** - プロトタイプを納品、体験をガイド

**出力**: `prototype/`（プロセス駆動ワークベンチ付きインタラクティブプロトタイプ、体験ヒント機能）

### プロセス管理

- **レジューム**: 中断後、最後のチェックポイントから再開
- **設計イテレーション**: 設計中およびプロトタイプ納品後の変更をサポート、自動影響評価
- **変更レベル**: L1 軽微な調整 → L4 重大な変更、それぞれロールバック範囲が異なる

## コアコンセプト（非技術言語）

| FBS層 | 技術用語 | 私たちの言い方 | 例 |
|-------|---------|--------------|---|
| F 機能層 | アクター | ユーザーロール | 管理者、購買担当、顧客 |
| F 機能層 | シナリオ | 使用シナリオ | 日常入庫、月次棚卸 |
| F 機能層 | プロセス | ビジネスプロセス | 注文から配送までの全プロセス |
| S 構造層 | ビジネスオブジェクト | 管理するもの | 顧客、注文、製品 |
| S 構造層 | 属性 | 情報/フィールド | 名前、電話、数量 |
| S 構造層 | 関係 | 関連/接続 | 顧客が複数の注文を「持つ」 |
| B 振る舞い層 | 状態 | 現在の状況 | 支払い待ち、発送済み |
| B 振る舞い層 | 操作 | できること | 承認、発送、キャンセル |
| B 振る舞い層 | ビジネスルール | 制約 | 管理者のみが承認可能 |

## 設計原則

1. **ユーザーファースト** - ユーザーのニーズと理解を技術的完璧さより優先
2. **プログレッシブディスクロージャー** - 徐々に深め、情報過多を避ける
3. **即時フィードバック** - 各ステップの後に即座に結果を表示
4. **透明で制御可能** - ユーザーが常に会話の方向を制御
5. **トレーサブル** - すべての決定と変更を記録

## 品質基準

### 完全性
- ✅ すべての必須フィールドが入力済み
- ✅ エンティティ属性≥2
- ✅ 明確な関係定義
- ✅ コア操作にビジネスルールあり

### 一貫性
- ✅ 統一された命名規則
- ✅ 循環依存なし
- ✅ 競合するビジネスルールなし

### 信頼度
- 90%-100%: 優秀 - 次のステップに進む
- 70%-90%: 良好 - 継続するかどうか質問
- 50%-70%: 普通 - さらなる明確化を推奨
- <50%: 低い - 重要な問題を解決する必要あり

## 変更管理

設計中およびプロトタイプ納品後の変更をサポートします。詳細は `references/iteration-rules.md` を参照してください。

4レベルの変更をサポート:

| レベル | タイプ | 例 | 処理方法 |
|-------|------|---|---------|
| L1 | 軽微な調整 | 表示名を変更 | 現在のステップで直接変更 |
| L2 | 局所的な修正 | 属性タイプを変更 | 関連ステップに戻る |
| L3 | 中程度の変更 | 新しいエンティティを追加 | フェーズの開始点に戻る |
| L4 | 重大な変更 | エンティティを削除 | 初期設計に戻るか新規プロジェクト作成 |

**変更前**: 影響分析を表示
**変更後**: 一貫性を検証、モデルとプロトタイプを同期

## 設計思想

このプロジェクトは以下の方法と概念から着想を得ています:

- **FBS存在論（Gero, 1990）** - Function-Behavior-Structure、モデリングフレームワークの理論的基盤
- **UML（統一モデリング言語）** - ユースケース図（F層）、クラス図（S層）、状態図（B層）
- **ドメイン駆動設計（DDD）** - ドメインオブジェクト、集約、ドメインイベント
- **Harness Engineering（2026）** - 品質駆動エージェント制約システム
- **ユーザーストーリーマッピング** - シナリオ駆動要件分析
- **会話型デザイン** - 自然言語インタラクションデザイン

詳細: `docs/core/DESIGN.md`

## 技術的制約

| 制約 | 影響 | 軽減策 |
|-----|------|--------|
| コンテキスト長 | 長い対話が制限を超える可能性 | セグメント化された読み込み、ファイル永続化 |
| ファイル並行性 | 複数セッションが競合する可能性 | ファイルロックメカニズム |
| プロトタイプ複雑さ | HTML/JSでバックエンドロジックを実装できない | プロトタイプの境界を明確化 |
| バージョン互換性 | YAML形式の変更 | バージョン番号管理 |
