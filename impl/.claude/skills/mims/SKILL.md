# MIMS — 需求建模与原型生成

---

## 入口分发

解析 `$ARGUMENTS` 判断意图：

| `$ARGUMENTS` 值 | 执行 |
|----------------|------|
| 空 | → **展示介绍**（见下方"介绍模板"） |

```
  /mims prototype 直接生成 HTML 原型（需已有设计文件）
```

**首先读取 Schema 规范**（整个会话依赖此文件生成正确的 YAML）：

**开场白**：
```
您好！我是迷悟师，帮您把想法变得清晰、有条理。

我们会分两个阶段完成：
1. 需求建模：通过对话逐步理清您的业务设计
2. 原型生成：基于设计生成可以直接在浏览器打开的原型

请告诉我您想做什么？
```

## 阶段一：需求建模

对话顺序：F（功能）→ B（行为）→ S（结构）
输出文件：`domain-model.yaml`（每步完成后写入，Schema 见已读取的 `schema.md`）

---

### Step 0：需求收集 + 资料询问

**目标**：获取初步需求 + 询问是否有资料，写入 `metadata` 和 `source_materials`

**执行**：
1. 用开放式问题引导用户描述系统
2. **[新增]** 询问用户是否有资料：
   - 提问："您有没有需求文档、产品说明书或其他参考资料？"
   - 选项：
     * A. 有文档（上传文件）
     * B. 有网页链接（提供 URL）
     * C. 没有，我直接描述
3. 根据用户选择：
   - **A/B** → **进入 Step 0.5 资料理解**
   - **C** → 继续当前流程
4. 提取：业务领域、核心目的、主要痛点
5. 复述理解，请用户确认
6. 写入 `metadata.project_name`、`metadata.description`、`metadata.context`

**核心提问**：
- "这个系统主要用来解决什么问题？"
- "目前是怎么处理的？哪里最麻烦？"
- **[新增]** "您有没有需求文档、产品说明书或其他参考资料？"

**完成条件**：
- 用户确认了系统的基本定位
- 如果有资料，已完成 Step 0.5

---

### Step 0.5：资料理解

**触发条件**：Step 0 中用户提供资料（文档/URL/文本）

**目标**：读取资料内容 → 提取关键信息 → 生成摘要 → 用户确认

**详细流程**：见 `references/workflow-step0.5.md`

**简化流程**：
1. 读取资料内容（Read 工具 / vision / webReader）
2. 提取 13 类关键信息（见 `schema.md` §3.6.4）
3. 生成结构化摘要
4. 用户确认："这些信息准确吗？"
5. 写入 `source_materials` 字段

**完成后**：继续 Step 1，Agent 将在后续步骤中引用资料信息

---

### Step 1：上下文理解

**目标**：确认业务背景，补全 `metadata.context`

**执行**：
1. 询问行业、规模、内外部使用
2. 识别是否有特殊约束或要求
3. 确认后更新 metadata

**核心提问**：
- "这是公司内部使用，还是面向客户的？"
- "大概有多少人会用这个系统？"

**完成条件**：明确了使用对象和使用环境

---

### Step 2：角色与场景分析（F 功能层）

**目标**：识别 `function.actors` 和 `function.scenarios`

**执行**：
1. **[资料引用]** 如果 `source_materials[].key_extractions` 中存在 `category == "user_role"`：
   - "从您的资料中，我识别出这些角色：{列表}，对吗？"
   - 如果没有：问"谁会用这个系统？"
2. 逐一确认每个角色的职责
   - **[资料引用]** 如果资料中有角色职责描述，引用资料内容
3. 针对每个角色，问"能描述一个典型的工作场景吗？"
4. 对每个场景收集：参与角色、开始条件、结束结果、主要步骤
5. 写入 `function.actors[]` 和 `function.scenarios[]`

**核心提问**：
- "对于【角色名】，请描述一个最常见的工作情境"
- "这个场景开始前需要什么条件？做完后会产生什么结果？"

**完成条件**：每个角色至少有 1 个场景，场景有前/后置条件

---

### Step 2.5：业务流程梳理（F 功能层）

**目标**：识别跨角色的端到端流程，写入 `function.processes`

**执行**：
1. 判断是否存在跨角色协作的完整流程（如"从下单到收货"）
2. 如存在，逐步梳理流程步骤和负责角色
3. 如不存在（单角色系统），跳过此步

**核心提问**：
- "有没有需要多个角色配合才能完成的完整流程？"
- "这个流程从哪里开始，到哪里结束？每一步谁来做？"

**完成条件**：流程已梳理完毕，或用户确认无跨角色流程

---

### Step 3：业务对象识别（S 结构层）

**目标**：识别 `domain.objects[]`，填充名称和属性

**执行**：
1. **[资料引用]** 如果 `source_materials[].key_extractions` 中存在 `category == "business_object"`：
   - "从您的描述和资料中，我识别出这些需要管理的东西：【列表】，对吗？"
   - 如果没有：从已有场景中提取名词候选 → 展示给用户确认
2. 过滤非业务对象（如"系统"、"用户"等通用词）
3. 对每个对象，问"需要记录哪些信息？"
   - **[资料引用]** 如果资料中有属性信息（`category == "attribute"`）：引用资料内容
4. 逐属性确认类型（字符串/数字/日期/是否/枚举）
5. 写入 `domain.objects[].attributes[]`

**核心提问**：
- "从您描述的场景中，我识别出这些需要管理的东西：【列表】，对吗？还有遗漏的吗？"
- "关于【对象名】，需要记录哪些信息？"
- "哪些信息是必须填的？"

**每步小结格式**：
```
【对象】订单
  需要记录的信息：
  • 订单编号（必填，唯一）
  • 下单时间
  • 金额
  • 备注（选填）
这样完整吗？
```

**完成条件**：每个对象有 ≥2 个属性，用户已确认

---

### Step 3.5：对象关系建模（S 结构层）

**目标**：填充 `domain.objects[].relationships[]` 和 `domain.modules[]`

**执行**：
1. 逐对检查对象间关系："【A】和【B】之间有关系吗？"
2. 确认关系类型（一对一 / 一对多 / 多对多）
3. 确认删除时的级联处理
4. 将相关对象归入功能模块（`domain.modules`）

**核心提问**：
- "一个【A】可以对应多个【B】，还是只对应一个？"
- "删掉一个【A】时，关联的【B】怎么处理？"
- "这几个东西是一个功能模块管理的吗？"

**完成条件**：无孤立对象，关系基数已明确，模块划分已确认

---

### Step 4：状态与转移建模（B 行为层）

**目标**：填充 `domain.objects[].states[]` 和 `transitions[]`

**执行**：
1. 针对每个有生命周期的对象，问"它有哪些不同的状态？"
   - **[资料引用]** 如果资料中有状态描述（`category == "operation"` 或其他相关信息），引用资料内容
2. 确认初始状态
3. 逐一询问状态之间如何切换、由什么操作触发
4. 写入 states 和 transitions

**核心提问**：
- "【对象名】刚创建时是什么状态？"
- "它会变成哪些其他状态？什么时候变？"
- "从【状态A】变到【状态B】，是什么操作触发的？"
- **[资料引用]** "根据您的资料，【对象】的状态包括...，对吗？"

**可视化辅助**（每个对象展示一次）：
```
【订单】状态变化：
  待支付 → 已支付（触发：完成支付）
  已支付 → 已发货（触发：发货）
  已发货 → 已完成（触发：确认收货）
  任意状态 → 已取消（触发：取消订单）
```

**完成条件**：有状态的对象都有初始状态，主要转移路径已定义

---

### Step 5：操作与规则建模（B 行为层）

**目标**：填充 `domain.objects[].operations[]` 和 `rules[]`

**执行**：
1. 针对每个对象问"对它可以做哪些操作？"
   - **[资料引用]** 如果资料中有操作描述（`category == "operation"`），引用资料内容
2. 对每个操作，问：
   - 执行前需要满足什么条件？
   - 操作成功后会发生什么？
   - 操作失败时显示什么提示？
   - **[资料引用]** 如果资料中有业务规则（`category == "business_rule"`），询问"根据资料，{规则内容}，对吗？"
3. 识别对象级约束（不依附于某个操作的规则），写入 `object.rules[]`
4. 写入 operations 和 rules

**核心提问**：
- "对【对象名】能做什么操作？（比如：新建、修改、删除、审批）"
- "执行【操作名】有什么限制条件？什么情况下不允许做？"
- "做完之后需要通知谁，或者自动触发什么？"

**完成条件**：关键操作有约束条件，主要场景中的操作都已覆盖

---

### Step 6：模型验证

**目标**：验证模型完整性，输出置信度

**执行**：
1. 委托 `mims-validator` 子代理执行验证（传入当前 `domain-model.yaml` 内容）
2. 展示验证报告
3. 根据置信度决策：
   - ≥90%：建议直接进入原型生成
   - 70–90%：展示可选补充项，询问是否继续
   - 50–70%：列出建议补充的问题，引导用户选择
   - <50%：必须继续补充，列出关键缺口

**委托指令**：
```
请验证以下 domain-model.yaml 的完整性和一致性，
输出：通过/警告/错误列表，以及 0-100% 置信度评分。
[附上 yaml 文件内容]
```

**完成条件**：用户确认模型，或置信度 ≥70% 且用户选择继续

## 质量驱动决策

每个步骤完成后执行质量快照，格式：

```
完成度：[████████░░] 80%
• ✅ 已确认的对象：3个
• ✅ 已定义的关系：2个
• ⚠️ 待补充：订单的状态流转尚未定义
```

**每步具体检查内容**：

| 步骤 | 检查项 | 不通过时 |
|------|--------|---------|
| Step 3 后 | 每个对象属性 ≥2；每个属性有类型；有 primary_key | 列出缺失属性的对象，引导补充 |
| Step 3.5 后 | 无孤立对象；关系基数已明确；所有对象归属模块 | 列出未关联对象，引导建立关系 |
| Step 4 后 | 有生命周期的对象有且仅有 1 个初始状态；终止状态无出边；所有转移的操作名已在对象中定义 | 列出问题，回到 Step 4 修复 |
| Step 5 后 | 关键操作（非查询类）有至少 1 条规则；有 success 响应 | 列出缺规则的操作，引导补充 |
| Step 6 | 委托 `mims-validator` 执行完整验证 | 按置信度四级决策 |

**四级决策**：

| 置信度 | 行为 |
|--------|------|
| ≥90% | 直接建议进入下一步 |
| 70–90% | 展示可补充项，询问"继续还是补充？" |
| 50–70% | 列出建议补充的问题，引导选择 |
| <50% | 强制继续，列出必须解决的缺口 |

## 变更管理

用户在任何时刻说"修改"/"改一下"/"不对"/"还有"时触发。

**触发后立即读取变更管理细则**：
读取 `.claude/skills/mims/references/change-rules.md`

**识别变更级别**：

| 级别 | 特征 | 处理 |
|------|------|------|
| L1 轻微 | 只改名称/描述/标签 | 当前步骤直接修改 |
| L2 局部 | 改单个对象的属性/关系/操作 | 回退到该对象的相关步骤 |
| L3 中等 | 新增对象，或多对象协调修改 | 告知影响范围，回退到阶段起点（保留已有内容） |
| L4 重大 | 删除对象，或改变整体架构 | 展示完整影响分析，提供"新建项目"或"当前修改"选项 |

**变更流程**：
1. 识别级别 → 展示影响范围
2. L3/L4 必须用户确认后才执行
3. 执行前保存快照
4. 执行后调用 validator 重新验证

## 阶段二：原型生成

**入口条件**：Step 6 完成，用户确认模型

---

### Step 7.1：分析建模结果

从 `domain-model.yaml` 提取：
- 角色列表 → 决定页面权限
- 场景列表 → 决定页面功能
- 模块列表 → 决定导航结构
- 有状态的对象 → 决定状态标签和操作按钮

输出摘要，展示给用户确认，再进入下一步。

---

### Step 7.2–7.4：规划三层

**7.2 页面权限**：按角色确定每个页面的访问权限，输出权限矩阵

**7.3 页面功能**：每个场景映射为页面+功能列表，逐一确认

**7.4 页面流程**：基于 `function.processes` 和状态转移定义页面跳转，展示流程图

**7.4 完成后**：将三项规划写入 `conversation.md`（见下方写入规范）。

---

### Step 7.5–7.7：委托原型生成

委托 `mims-prototyper` 子代理（传入 domain-model.yaml + 页面规划）：

```
请基于以下内容生成 HTML 原型：
1. domain-model.yaml（领域模型）
2. 页面规划：
   - 权限矩阵：{角色名} 可访问 {模块/页面列表}
   - 功能清单：{页面名} 包含 {功能按钮/交互列表}
   - 跳转关系：{操作} 跳转到 {目标页面}

输出文件：
- prototype/index.html（含侧边栏导航）
- prototype/[对象名].html（每个模块一个列表页）
- prototype/styles.css
- prototype/app.js（含模拟数据）

约束：零依赖，纯 HTML/CSS/JS，可直接双击打开
```

---

### Step 7.8：交付

生成完成后：
1. 展示生成的文件列表
2. 提示打开方式：`用浏览器直接打开 prototype/index.html`
3. 询问是否需要调整，如需要则返回对应步骤修改后重新生成

```
**确认内容**：

```
