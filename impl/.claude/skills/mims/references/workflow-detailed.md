# 详细设计 D1–D5：详细执行流程

> 本文档描述详细设计阶段各步骤的执行流程，由 SKILL.md 引用。
>
> **入口条件**：初步设计 P6 完成，检查点通过
> **覆盖层**：S 结构层 + B 行为层
> **目标读者**：迷悟师 Agent（非用户）

---

## D1：业务对象识别（S 结构层）

**目标**：识别 `domain.objects[]`，填充名称和属性。

**执行流程**：

1. 从 P4 场景和 P5 流程中提取名词候选 → 展示给用户确认
2. **过滤非业务对象**：排除以下类型的名词：
   - 通用系统词：「系统」「平台」「软件」「程序」「模块」「功能」
   - 通用角色词：「用户」「人员」「所有人」（已被 actors 覆盖）
   - 抽象概念：「流程」「规则」「权限」「数据」「信息」
   - 动作词：「操作」「管理」「查询」「审批」（应归入 operations）
   判断原则：如果该名词代表一个需要**独立存储和管理**的业务实体（有自己的属性和生命周期），则保留为对象。
3. 如果 `source_materials[].key_extractions` 存在 `category == "business_object"`，优先引用
4. 逐对象问："关于【对象名】，需要记录哪些信息？"
5. 逐属性确认类型：字符串 / 数字 / 日期 / 是否 / 枚举 / 引用
6. 写入 `domain.objects[].attributes[]`，将对象归入 P6 定义的模块

**对话示例**：

```
Agent：从您描述的场景中，我识别出这些需要管理的东西：
  1. 订单  2. 商品  3. 客户
  还有遗漏的吗？

用户：还有收货地址。

Agent：好的，一共 4 个。先看「订单」——需要记录哪些信息？

用户：订单号、下单时间、金额、状态。

Agent：确认一下：
  • 订单号 — 唯一标识？类型：字符串
  • 下单时间 — 类型：日期
  • 金额 — 类型：数字
  • 状态 — 有固定选项吗？还是自由填写？
```

**完成条件**：每个对象 ≥ 2 个属性，每个属性有类型，每个对象已归属模块。

---

## D2：对象关系与模块归属（S 结构层）

**目标**：填充 `domain.objects[].relationships[]`，确认 `domain.modules[]` 归属。

**执行流程**：

1. 逐对检查关系："【A】和【B】之间有关联吗？"
2. 确认关系类型：一对一 / 一对多 / 多对多
3. 确认级联处理："删掉一个【A】时，关联的【B】怎么处理？"
4. 验证 P6 每个模块至少有 1 个对象
5. 写入 relationships 和最终模块归属

**对话示例**：

```
Agent：一个「订单」可以对应几个「商品」？

用户：一个订单可以买多个商品。

Agent：一对多的关系。删掉订单时，里面的商品记录怎么处理？

用户：一起删掉。

Agent：好的。确认一下模块划分：
  • 订单管理：订单、商品
  • 客户管理：客户、收货地址
  合适吗？
```

**完成条件**：无孤立对象，关系基数已明确，每个模块 ≥ 1 个对象（`is_ai_agent=true` 的模块豁免此检查），AI Agent 模块设计已确认（如有）。

---

## D2.5：跨阶段交叉验证

**目标**：在对象和模块归属确定后，执行 F↔S/B 交叉检查，确保初步设计（F 层）与详细设计（S/B 层）之间的一致性。

**执行流程**：

1. 委托 `mims-validator` 执行跨阶段验证（mode: cross-stage），重点检查：
   - **场景落地性**：每个场景的 workflow 步骤是否都有对应的对象/操作支撑
   - **模块覆盖**：P6 推断的模块是否与 D2 实际归属的模块一致
   - **流程步骤归属**：P5 流程步骤中的 module 字段是否与 D2 模块名一致（P5 的自由文本需在此处映射为正式模块名）
2. 如有 ERROR → 引导用户修复后再进入 D3
3. 如有 WARNING → 展示给用户，询问是否调整

**完成条件**：跨阶段验证无 ERROR，所有 WARNING 已处理或确认接受。

### D2 补充：AI Agent 模块设计确认

在 D2 完成模块归属验证后，检查所有 `is_ai_agent: true` 的模块。对每个 AI Agent 模块，逐一引导用户确认以下 6 项：

**1. 总体功能**：确认这个 Agent 的核心职责

```
【{模块中文名}】是一个 AI Agent 模块。
我们来确认它的设计细节。

首先，它的总体功能是什么？
之前您提到：{P6 阶段的 ai_justification}
请再具体描述一下，您期望这个 Agent 做什么？
```

**2. Agent 人设**：定义 Agent 的身份和行为边界

```
这个 Agent 在与用户交互时，应该是什么样的身份？
比如：专业的客服助手、友好的学习伙伴、严谨的审核员...

它的性格、语气、行为边界是什么？
有什么是它绝对不能做的？
```

**3. 技能规划**：Agent 需要具备的能力

```
这个 Agent 需要哪些技能？比如：
• 理解自然语言并提取关键信息
• 基于规则生成推荐方案
• 查询数据库并整理结果
• 多轮对话引导用户完成流程

每个技能具体做什么？什么情况下触发？
```

**4. MCP（外部工具）规划**：Agent 需要调用的外部能力

```
这个 Agent 是否需要调用外部工具或系统？比如：
• 查询外部数据（如天气、物流信息）
• 调用内部模块的功能（如库存查询）
• 发送通知（如邮件、短信）
• 读写文件

每个工具的用途和调用时机是什么？
```

**5. 协作关系**：与其他模块和外部系统的关系

```
最后确认一下这个 Agent 与其他部分的关系：
• 它需要从哪些模块获取数据？
• 它的处理结果会传给哪些模块？
• 它是否依赖外部系统（已在 P6 外部接口中定义的）？
• 用户通过什么方式与它交互（对话界面、嵌入页面、独立入口）？
```

**6. 对话示例**：基于以上 5 项，自动生成典型对话示例供用户确认

Agent 根据已确认的人设、技能、工具等信息，自动生成 2-3 个典型对话示例，展示 Agent 如何在实际场景中与用户交互。每个示例覆盖一种核心技能或典型场景。

```
基于我们刚才确认的设计，我生成了几个典型对话示例，
看看是否符合您的预期：

【示例 1：{场景名称}】
用户：{模拟用户输入}
Agent：{模拟 Agent 回复，体现人设语气和技能运用}

【示例 2：{场景名称}】
用户：{模拟用户输入}
Agent：{模拟 Agent 回复}

这些对话的语气和回答方式，是您期望的吗？
需要调整哪些方面？
```

用户确认后，对话示例作为该 Agent 的行为参考，写入 SDD。

## D3：状态与生命周期（B 行为层）

**目标**：填充 `domain.objects[].states[]` 和 `transitions[]`。

**执行流程**：

1. **判断是否需要状态**：
   - 有明确生命周期变化的对象（如订单有"待支付→已支付→已发货"）→ 需要状态定义
   - **纯静态对象**（满足以下全部条件）可跳过：
     - 只有增删改查操作，无状态流转
     - 创建后内容基本不变（如城市字典、分类标签）
     - 没有审核、审批等流程性操作
   - 示例：地址簿通常是静态的（创建后不变），但如果地址有"默认/非默认"切换则不是纯静态
2. 问："这个东西有哪些不同的状况？"
3. 确认初始状态（每个对象必须有且仅有 1 个）
4. 问："从【状态A】变到【状态B】，是什么操作触发的？"
5. 可视化展示状态变化路径

**对话示例**：

```
Agent：「订单」从创建到结束，会经历哪些不同的状况？

用户：待支付、已支付、已发货、已完成、已取消。

Agent：确认一下变化路径：
  待支付 → 已支付（触发：完成支付）
  已支付 → 已发货（触发：发货）
  已发货 → 已完成（触发：确认收货）
  任意状态 → 已取消（触发：取消订单）
  有遗漏的路径吗？
```

**完成条件**：有状态的对象有初始状态，主要转移路径已定义，终止状态无出边。

## D4：操作与业务规则（B 行为层）

**目标**：填充 `domain.objects[].operations[]` 和 `rules[]`。

**执行流程**：

1. **遍历所有对象**：依次对 `domain.objects[]` 中每个对象执行操作识别，确保每个对象至少识别出基本的 CRUD 或业务操作，不能只处理部分对象
2. 问："对【对象名】能做什么操作？"
3. 如果资料中有 `category == "operation"` 或 `"business_rule"` 的提取项，引用
4. 逐操作确认：执行条件 / 成功后发生什么 / 失败提示
5. 识别对象级约束（不依附于特定操作的规则）
6. 写入 operations 和 rules
7. **回填 permissions**：D4 完成后，根据每个角色（`function.actors[]`）的场景归属和操作归属，回填 `function.actors[].permissions`（列出该角色可执行的操作名）

**对话示例**：

```
Agent：对「订单」能做哪些操作？

用户：创建订单、支付、取消。

Agent：逐个确认。先看「支付」：
  • 条件：必须是"待支付"状态？
  • 成功后：自动变"已支付"，还需要通知谁？
  • 失败时提示什么？

用户：失败提示"支付未成功，请重试"。成功后通知仓库准备发货。

Agent：记下了。另外，有没有整体约束？比如"金额不能超过 10 万"？

用户：有的，单笔订单上限 10 万。

Agent：好的，这条作为订单的整体约束记录。
```

**完成条件**：所有对象都已识别操作；关键操作有 ≥ 1 条规则，有 success 响应，对象级约束已记录；permissions 已回填。

## D5：模型验证与置信度评估

**目标**：验证模型完整性，输出置信度，生成详细设计检查点。

**执行流程**：

1. 委托 `mims-validator` 子代理（mode: full），传入 `domain-model.yaml`
2. 展示验证报告：通过/警告/错误 + 置信度评分
3. 根据置信度决策：

| 置信度 | 处理 |
|--------|------|
| ≥ 90% | 建议进入原型生成 |
| 70–90% | 展示可选补充项，询问是否继续 |
| 50–70% | 列出建议补充的问题，引导用户 |
| < 50% | 必须继续补充 |

4. **验证未通过时的引导**：
   - 置信度 50-70%：列出具体缺口，用自然语言逐项请用户补充。"我检查了一下，发现还需要补充以下内容：{逐条列出}。我们先从第一个开始？"
   - 置信度 <50%：列出必须解决的问题，引导用户逐个解决。"要继续往下走，有几个关键问题需要先解决：{逐条列出}。"
   - 每次补充后重新验证，不跳过未解决的问题。

5. **详细设计检查点**：
   - 置信度达标 → 加载 `references/sdd-template.md`，委托 mims-spec-generator（type: sdd），传入模板内容生成 `sdd.md`
   - 展示摘要：对象数、关系数、状态数、操作数、规则数
   - 用户确认 → 展示阶段过渡引导（见 `workflow-common.md` §五 D5→R1）→ 进入原型生成

   - 更新 `metadata.design_progress.detailed.status = "complete"`

**检查点摘要示例**：

```
详细设计完成：
  • 要管理的东西：4 个
  • 关联关系：3 条
  • 有状态的：1 个（5 个状态）
  • 可以做的操作：7 个
  • 约束条件：4 条
  • 置信度：88%
  已生成：sdd.md
  是否进入原型生成？
```

**委托指令**：

```
验证 domain-model.yaml 完整性和一致性。
输出：通过/警告/错误 + 0-100% 置信度。
重点：主键、孤立对象、状态闭环、操作覆盖场景。
[附 yaml 内容]
```

**完成条件**：验证报告已展示，置信度 ≥ 70% 或用户选择继续，`sdd.md` 已生成，`design_progress.detailed.status = "complete"`。
