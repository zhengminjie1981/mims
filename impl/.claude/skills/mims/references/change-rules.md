# 变更管理

## 触发条件

以下两种方式均可触发变更管理流程：

**用户主动触发**：用户说"修改"/"改一下"/"不对"/"还有"等词语时。

**Agent 检测触发**：Agent 在建模过程中检测到以下问题时主动提出：
- 发现新对象与已有对象存在未声明的关系
- 检测到规则冲突（两条操作规则互相矛盾）
- 发现引用断裂（某操作/场景引用了不存在的对象或状态）
- 用户描述与已建模内容矛盾

Agent 检测触发时，先向用户说明问题，再询问是否进入变更流程。

---

## 变更级别与处理

| 级别 | 判断标准 | 处理动作 |
|------|---------|---------|
| **L1 轻微** | 只改名称、描述、标签、中文名 | 当前步骤直接修改，无需回退 |
| **L2 局部** | 改单个对象的属性/关系/操作/规则 | 回退到该对象所在的建模步骤 |
| **L3 中等** | 新增对象，或多对象协调修改 | 回退到阶段一起点，保留已有内容供参考 |
| **L4 重大** | 删除对象，或改变整体架构/核心场景 | 展示完整影响分析，提供"新建项目"或"当前重建"两个选项 |

---

## 执行流程

1. **识别级别** — 判断 L1/L2/L3/L4，向用户说明
2. **变更前验证** — 检查以下问题，若存在则一并告知用户：
   - 变更后是否产生孤立对象（无任何关系的对象）
   - 是否有关系因对象删除而断裂
   - 是否有操作/规则引用了将被删除的对象或状态
   - 是否有场景因对象/操作变更而无法执行
3. **展示影响** — 列出受影响的对象、关系、操作、场景（见下方"展示影响"格式）
4. **获取确认** — L3/L4 必须用户明确确认后才执行；L1/L2 可直接执行
5. **保存快照** — 执行前将当前 `domain-model.yaml` 内容追加到 `conversation.md`（注明版本和变更原因）
6. **执行修改** — 按回退动作修改模型
7. **重新验证** — 委托 `mims-validator` 验证修改后的模型

---

## 展示影响

展示影响时，逐条列出受影响的：
- 对象（新增/删除/修改）
- 关系（需要调整的）
- 操作和规则（失效或需更新的）
- 场景（需要更新的）
- 原型（是否需要重新生成）

---

## 快照时机

| 时机 | 动作 |
|------|------|
| 每个建模步骤完成时 | 将当前 `domain-model.yaml` 写入磁盘（覆盖更新） |
| 变更执行前（步骤 5） | 将 yaml 内容追加到 `conversation.md`，注明版本号和变更原因 |
| 阶段一全部完成时 | 在 `conversation.md` 追加完整模型快照，标记为里程碑 |

---

## 变更记录格式

每次变更后追加到 `conversation.md`：

```markdown
## 变更记录 #{n} — {datetime}

- 变更级别：L{n}（{轻微/局部/中等/重大}）
- 触发方式：{用户主动 / Agent检测}
- 变更内容：{一句话描述修改了什么}
- 影响范围：{受影响的对象/关系/操作列表}
- 用户确认：是
```
