# 原型生成详细流程文档（R1–R9）

> 本文档详细描述阶段二"原型生成"的执行流程，由 SKILL.md 引用。
>
> **触发条件**：Step 6（模型验证）完成，用户确认模型
> **目标**：将 domain-model.yaml 映射为零依赖 HTML/CSS/JS 可交互原型
> **前置依赖**：`references/schema.md` 已加载

---

## 术语说明

原型阶段涉及两个不同层次的"功能"，需明确区分：

| 术语 | 含义 | 来源 | 示例 |
|------|------|------|------|
| **业务功能** | 系统的业务目的，"做什么"和"为什么做" | FBS F 层（场景、流程） | "员工提交报销单"、"主管审批" |
| **页面功能** | 用户在页面上的交互操作，"怎么操作" | 原型阶段（R3） | "提交"按钮、"通过/驳回"按钮 |

**对应关系**：流程中的**用户交互步骤**（非系统自动执行的步骤）可对应一个或多个页面功能。这是隐式对应，非强制映射——流程步骤可能包含一系列操作及规则，不要求逐一对应页面功能。

---

## 流程概览

```
R1 分析建模结果
 ↓
R2 页面权限
 ↓
R3 页面功能映射
 ↓
R4 页面流程设计
 ↓
R5 页面结构设计
 ↓
R6 页面交互设计
 ↓
R7 代码生成（委托子代理）
 ↓
R8 流程验证
 ↓
R9 交付
```

---

## R1：分析建模结果

**目标**：从 domain-model.yaml 提取原型所需的全部输入数据

**提取清单**：

| 数据来源 | 提取内容 | 用途 |
|---------|---------|------|
| `function.actors[]` | 角色列表 | → 页面权限（R2） |
| `function.scenarios[]` | 场景列表 | → 页面功能（R3） |
| `domain.modules[]` | 模块列表 | → 导航结构（R5） |
| `domain.objects[]` 有 states 的对象 | 状态与转移 | → 状态标签、操作按钮可见性 |
| `function.processes[]` | 业务流程列表 | → 判断是否需要工作台页面 |

**流程判断**：若 `function.processes` 存在且非空，标记 `needs_workbench = true`。

**输出摘要格式**：

```
建模结果分析：

| 维度 | 数量 | 详情 |
|------|------|------|
| 用户角色 | {n} 个 | {角色名列表} |
| 使用场景 | {n} 个 | {场景名列表} |
| 功能模块 | {n} 个 | {模块名列表} |
| 业务对象 | {n} 个 | {对象名列表} |
| 有状态对象 | {n} 个 | {对象名：状态数} |
| 业务流程 | {n} 条 | {流程名列表} |

{如果有流程} 检测到跨角色流程，将生成工作台页面作为流程枢纽。
{如果没有流程} 未检测到跨角色流程，按模块独立生成页面。

以上分析正确吗？
```

**流程步骤与页面功能对应分析**（当存在 `function.processes` 时）：

对每个流程，列出其用户交互步骤（`system: false` 或未标记 system 的步骤），为 R3 页面功能映射提供参考：

```
流程步骤与页面功能对应：

【{流程名}】（场景：{父场景名}）
  步骤1：{actor} {action}（用户交互）→ 涉及模块：{module}，可能对应页面功能
  步骤2：系统自动 {action}（无需页面操作）
  步骤3：{actor} {action}（用户交互）→ 涉及模块：{module}，可能对应页面功能
  ...

【{流程名}】（场景：{父场景名}）
  步骤1：...
```

此分析为 R3 的页面功能映射提供输入，**不要求强制一一对应**。

**完成条件**：用户确认摘要

---

## R2：页面权限

**目标**：按角色确定每个模块/页面的访问权限

**执行**：

1. 遍历 `function.actors[].permissions`，将权限映射到模块
2. 若角色有某模块中任一对象的操作权限，则该角色可访问该模块页面
3. 输出权限矩阵供用户确认

**权限矩阵模板**：

```
页面权限矩阵：

| 角色 | {模块A页} | {模块B页} | 工作台 |
|------|----------|----------|--------|
| {角色1} | 可访问 | — | 可访问 |
| {角色2} | 可访问 | 可访问 | 可访问 |

需要调整哪些权限吗？
```

**完成条件**：用户确认权限矩阵

---

## R3：页面功能映射

**目标**：将场景和流程步骤映射到具体页面上的功能

**执行**：

1. 遍历 `function.scenarios[].workflow`，将每个 step 映射到对应模块页面的操作
2. 遍历 `function.processes[].steps`，将每个流程步骤映射到对应页面的操作按钮
3. 确保：流程中的每一步在原型中有对应的操作按钮或入口
4. 逐页面展示功能清单，逐一确认

**功能清单模板**：

```
{页面名} 功能清单：

| 来源 | 功能 | 操作类型 | 说明 |
|------|------|---------|------|
| 场景: {场景名} | {功能名} | 创建/查看/编辑/删除/业务操作 | {描述} |
| 流程: {流程名} Step{n} | {功能名} | 业务操作 | {流程步骤描述} |

这样完整吗？
```

**完成条件**：每个页面的功能清单经用户确认，流程覆盖检查通过或用户确认接受缺口

**流程覆盖检查**（页面功能确认后执行）：

逐流程检查用户交互步骤是否有对应的页面功能：

```
流程覆盖检查：

【{流程名}】（场景：{父场景名}）
  步骤1：{actor} {action}
    → 页面「{页名}」有「{页面功能}」✓
  步骤2：系统自动 {action} ✓
  步骤3：{actor} {action}
    → ⚠️ 未找到对应页面功能，是否需要补充？

{所有步骤均有对应} 所有流程步骤均可通过页面功能完成。
```

---

## R4：页面流程设计

**目标**：定义页面之间的跳转关系和导航路径

**执行**：

1. 基于 `function.processes` 的步骤顺序定义页面跳转链
2. 基于状态转移（`transitions[]`）定义操作后的跳转
3. 如存在跨角色流程，规划工作台页面作为流程枢纽
4. 展示流程图（文本形式）

**流程图模板**：

```
页面跳转关系：

【{流程名}】
  工作台 → {模块A页}（操作：{触发操作}）
  {模块A页} → {模块B页}（操作：{触发操作}，条件：{状态条件}）
  {模块B页} → 工作台（操作：完成后返回）

【通用导航】
  侧边栏 → 各模块页（始终可访问）

这个跳转关系正确吗？
```

**工作台设计**（仅当 `needs_workbench = true`）：
- 工作台作为所有流程的入口和进度追踪中心
- 从工作台可跳转到每个流程步骤对应的模块页面
- 完成操作后自动返回工作台查看进度

**完成条件**：用户确认跳转关系

---

## R5：页面结构设计

**目标**：为每个页面确定布局类型

**布局推荐规则**：

| 条件 | 推荐布局 | 典型场景 |
|------|---------|---------|
| 属性多（>8），数据结构化 | 表格列表 | 订单管理、客户列表 |
| 属性少（<=5），有图片/视觉元素 | 卡片视图 | 商品展示、员工名片 |
| 状态少（<=6个），强调流程推进 | 看板视图 | 任务管理、招聘流程 |
| 日期为核心维度 | 日历视图 | 会议预约、排班管理 |

**工作台布局**（固定）：任务看板 + 流程进度条 + 流程图

**结构方案模板**：

```
页面结构方案：

| 页面 | 要管理的东西 | 属性数 | 状态数 | 推荐布局 | 理由 |
|------|------------|-------|--------|---------|------|
| 工作台 | — | — | — | 工作台 | 跨角色流程枢纽 |
| {页面A} | {对象名} | {n} | {n} | {布局类型} | {理由} |

这个方案合不合适？有哪个页面想调整吗？
```

**完成条件**：每个页面有确定的布局类型，用户确认

---

## R6：页面交互设计

**目标**：确定每个页面的交互方式

**交互推荐规则**：

| 维度 | 条件 | 推荐方式 |
|------|------|---------|
| 详情查看 | 属性多/有编辑需求 | 模态框 |
| 详情查看 | 只读查看 | 侧边抽屉 |
| 新建方式 | 属性少（<=6） | 模态框 |
| 新建方式 | 属性多或分步 | 独立页面/分步向导 |
| 操作按钮 | 操作少（<=3） | 行内按钮 |
| 操作按钮 | 操作多（>3） | 顶部工具栏 |
| 关联数据 | 1:1 关系 | 行内展示 |
| 关联数据 | 1:N 关系 | 标签页子表格 |

**工作台交互**（固定）：点击任务卡片跳转模块页 + 任务卡片操作按钮 + 流程进度条

**交互方案模板**：

```
页面交互方案：

| 页面 | 查看详情 | 新建方式 | 操作位置 | 关联数据展示 |
|------|---------|---------|---------|------------|
| 工作台 | 点击任务跳转模块页 | — | 任务卡片操作按钮 | 流程进度+步骤列表 |
| {页面A} | {方式} | {方式} | {位置(数量)} | {方式} |

这个方案合不合适？需要调整哪些页面？
```

**完成条件**：每个页面有确定的交互方式，用户确认

**R6 完成后**：将权限矩阵 + 功能清单 + 跳转关系 + 结构方案 + 交互方案整体写入 `domain-model.yaml` 的 metadata 中（供 `/mims prototype` 命令复用），更新 `metadata.current_step = "R7"`。

---

## R7：代码生成

**目标**：委托 mims-prototyper 子代理生成零依赖 HTML 原型

**委托指令**：

```
请基于以下内容生成 HTML 原型：
1. domain-model.yaml（领域模型）
2. 页面规划：
   - 权限矩阵：{角色名} 可访问 {模块/页面列表}
   - 功能清单：{页面名} 包含 {功能按钮/交互列表}
   - 跳转关系：{操作} 跳转到 {目标页面}
3. 页面结构：{页面名} 使用 {布局类型}
4. 页面交互：{页面名} 详情={方式} 新建={方式} 操作={位置} 关联={方式}
5. 业务流程数据：{流程名：步骤列表，每步：操作、角色、对应模块页面}

输出文件：
- prototype/index.html（含侧边栏导航）
- prototype/workbench.html（工作台页面，仅当 needs_workbench=true 时生成）
- prototype/{对象名}.html（每个模块一个页面，按布局类型生成）
- prototype/{对象名}-form.html（新建方式为"独立页面"时生成）
- prototype/styles.css
- prototype/app.js（含模拟数据和流程模拟）

约束：零依赖，纯 HTML/CSS/JS，可直接双击打开
```

**完成条件**：子代理返回生成的文件列表

---

## R8：流程验证

**目标**：委托 mims-validator（mode: prototype）验证原型与模型的一致性

**执行**：
1. 收集 `prototype/` 目录中的 HTML 文件内容
2. 委托 mims-validator（mode: prototype），传入：
   - `domain-model.yaml` 完整内容
   - prototype 文件内容
   - 页面规划数据（权限矩阵、功能清单、跳转关系）
3. 展示验证报告

**问题处理**：
- 有 ERROR 项 → 返回 R3 补充缺失的操作按钮定义，重新生成受影响页面
- 全部通过或用户确认接受 → 进入 R9

**完成条件**：所有流程步骤验证通过，或用户确认接受当前结果

---

## R9：交付

**执行**：

1. 展示生成的文件列表：

```
已生成以下文件：

prototype/
├── index.html          ← 主入口（含侧边栏导航）
├── workbench.html      ← 工作台页面{仅当存在跨角色流程时}
├── {对象名}.html       ← {模块中文名}管理页
├── {对象名}-form.html  ← 新建表单页{如使用独立页面}
├── styles.css          ← 全局样式
├── app.js              ← 交互逻辑和模拟数据
└── feedback.md         ← 评审反馈文件（初始为空模板）
```

2. 提示打开方式和反馈收集：

```
原型已生成。您可以：

1. 在浏览器中打开 prototype/index.html 体验原型
2. 发现问题可直接在原型工作台的"评审反馈"区域记录，或点击页面底部的"记录反馈"快捷按钮
3. 反馈会自动保存到 prototype/feedback.md
4. 体验完成后告诉我，我会读取 feedback.md 中的反馈逐条处理

当然，您也可以直接口头告诉我需要调整的地方。
```

3. 收到反馈后，加载 `references/iteration-rules.md`，按简单/复杂反馈分类处理。如果存在 `prototype/feedback.md` 且非空，优先读取该文件作为反馈来源。

## 映射表与参考数据

以下表格供 Agent 在 R1–R6 各步骤中查阅，确保映射一致性。

### 1. 模型到界面映射

| 模型元素 | YAML 路径 | 界面元素 |
|---------|----------|---------|
| 对象 | `domain.objects[]` | 管理页面（一个模块一个页面） |
| 属性 | `objects[].attributes[]` | 表单字段 / 列表列 / 详情项 |
| 枚举属性 | `attributes[type=enum]` | 下拉选择 / 状态标签（带颜色） |
| 关联属性 | `attributes[type=reference]` | 选择器弹窗 / 自动补全 |
| 关系 1:1 | `relationships[cardinality=one_to_one]` | 详情页行内展示 |
| 关系 1:N | `relationships[cardinality=one_to_many]` | 标签页子表格 |
| 状态 | `objects[].states[]` | 状态标签（带颜色和图标） |
| 转移 | `objects[].transitions[]` | 操作按钮（按状态动态显示） |
| 操作 | `objects[].operations[]` | 页面按钮（位置由交互设计决定） |
| 业务规则 | `objects[].rules[]` | 表单校验提示 / 操作确认弹窗 |
| 角色 | `function.actors[]` | 页面访问权限 / 侧边栏可见性 |
| 场景 | `function.scenarios[]` | 页面功能集合 |
| 流程 | `function.processes[]` | 工作台流程图 / 进度追踪 |

### 2. 状态机映射模板

对每个有状态的对象，按此模板生成状态机数据：

```yaml
# 对象: {对象名}
state_machine:
  initial_state: "{is_initial=true 的状态名}"
  states:
    - name: "{状态名}"
      label: "{中文名}"
      color: "{UI色值}"    # 从 data_dictionary 或自动分配
      icon: "{图标名}"     # 可选
      is_final: {true/false}
  transitions:
    - from: "{源状态}"
      to: "{目标状态}"
      trigger: "{操作名}"
      trigger_label: "{操作中文名}"
      condition: "{附加条件，可选}"
```

### 3. 业务规则索引模板

按对象索引所有规则，供原型生成时实现校验逻辑：

```yaml
# 对象: {对象名}
business_rules:
  object_rules:        # 对象级规则（不变量）
    - id: "{rule_id}"
      constraint: "{约束表达式}"
      message: "{错误提示}"
      severity: "{error/warning}"
  operation_rules:     # 操作级规则（前置条件）
    - operation: "{操作名}"
      rules:
        - condition: "{条件}"
          action: "{ALLOW/DENY/WARN}"
          message: "{提示信息}"
```

### 4. 原型页面参考

| 原型文件 | 对应模型 | 生成条件 |
|---------|---------|---------|
| `prototype/index.html` | 全局导航 + 所有模块入口 | 必须生成 |
| `prototype/workbench.html` | `function.processes[]` | 仅当 processes 非空 |
| `prototype/{module}.html` | `domain.modules[]` + 所属 objects | 每个模块一个页面 |
| `prototype/{module}-form.html` | `objects[].attributes[]` | 仅当新建方式为"独立页面" |
| `prototype/styles.css` | 全局样式 | 必须生成 |
| `prototype/app.js` | 模拟数据 + 交互逻辑 + 流程模拟 | 必须生成 |
| `prototype/feedback.md` | 评审反馈记录 | 必须生成（空模板） |

### 5. 建议实现顺序

基于 `domain.modules[].depends_on` 依赖关系，按以下规则确定页面生成顺序：

1. **拓扑排序**：无依赖的模块先生成，被依赖的模块先生成
2. **无依赖关系时**：按对象数量从少到多排列
3. **工作台**：始终在最后生成（依赖所有模块页面的路由）

```
生成顺序示例：
1. {模块A}（无依赖，对象数少）
2. {模块B}（依赖模块A）
3. {模块C}（无依赖，对象数多）
4. 工作台（依赖所有模块页面）
5. index.html（依赖所有页面路由）
```
