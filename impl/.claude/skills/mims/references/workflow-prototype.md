# 原型生成详细流程文档（R1–R9）

> 本文档详细描述阶段二"原型生成"的执行流程，由 SKILL.md 引用。
>
> **触发条件**：Step 6（模型验证）完成，用户确认模型
> **目标**：将 domain-model.yaml 映射为零依赖 HTML/CSS/JS 可交互原型
> **前置依赖**：`references/schema.md` 已加载

---

## 术语说明

原型阶段涉及两个不同层次的"功能"，需明确区分：

| 术语 | 含义 | 来源 | 示例 |
|------|------|------|------|
| **业务功能** | 系统的业务目的，"做什么"和"为什么做" | FBS F 层（场景、流程） | "员工提交报销单"、"主管审批" |
| **页面功能** | 用户在页面上的交互操作，"怎么操作" | 原型阶段（R3） | "提交"按钮、"通过/驳回"按钮 |
| **评审工具** | 嵌入原型的设计核对辅助，帮助用户验证原型是否与设计一致 | 原型阶段（R5-R7） | 底部抽屉：场景/流程/对象/操作/规则/权限矩阵 |

**对应关系**：流程中的**用户交互步骤**（非系统自动执行的步骤）可对应一个或多个页面功能。这是隐式对应，非强制映射——流程步骤可能包含一系列操作及规则，不要求逐一对应页面功能。

**评审工具 ≠ 系统功能**：评审工具（需求抽屉）是嵌入每个原型页面的设计核对视图，用于非技术用户核对设计是否正确落地。它独立于业务逻辑，使用 `position: fixed` 不影响业务页面布局。工作台是业务功能页面，两者性质完全不同。

```

 ↓
R7 代码生成（委托子代理）
 ↓
R8 流程验证
 ↓
R9 交付
```

## R1：分析建模结果

**目标**：从 domain-model.yaml 提取原型所需的全部输入数据

**提取清单**：

| 数据来源 | 提取内容 | 用途 |
|---------|---------|------|
| `function.actors[]` | 角色列表 | → 页面权限（R2） |
| `function.scenarios[]` | 场景列表 | → 页面功能（R3） |
| `domain.modules[]` | 模块列表 | → 导航结构（R5） |
| `domain.objects[]` 有 states 的对象 | 状态与转移 | → 状态标签、操作按钮可见性 |
| `function.processes[]` | 业务流程列表 | → 判断是否需要工作台页面 |

**流程判断**：若 `function.processes` 存在且非空，标记 `needs_workbench = true`。

**输出摘要格式**：

```
建模结果分析：

| 维度 | 数量 | 详情 |
|------|------|------|
| 用户角色 | {n} 个 | {角色名列表} |
| 使用场景 | {n} 个 | {场景名列表} |
| 功能模块 | {n} 个 | {模块名列表} |
| 业务对象 | {n} 个 | {对象名列表} |
| 有状态对象 | {n} 个 | {对象名：状态数} |
| 业务流程 | {n} 条 | {流程名列表} |

{如果有流程} 检测到跨角色流程，将生成工作台页面作为流程枢纽。
{如果没有流程} 未检测到跨角色流程，按模块独立生成页面。
{始终} 原型每个页面将内置评审工具（底部抽屉），展示场景、流程、对象、操作、规则和权限矩阵，供您核对设计是否正确落地。

以上分析正确吗？
```

**流程步骤与页面功能对应分析**（当存在 `function.processes` 时）：

对每个流程，列出其用户交互步骤（`system: false` 或未标记 system 的步骤），为 R3 页面功能映射提供参考：

```
流程步骤与页面功能对应：

【{流程名}】（场景：{父场景名}）
  步骤1：{actor} {action}（用户交互）→ 涉及模块：{module}，可能对应页面功能
  步骤2：系统自动 {action}（无需页面操作）
  步骤3：{actor} {action}（用户交互）→ 涉及模块：{module}，可能对应页面功能
  ...

【{流程名}】（场景：{父场景名}）
  步骤1：...
```

此分析为 R3 的页面功能映射提供输入，**不要求强制一一对应**。

**完成条件**：用户确认摘要

## R2：页面权限

**目标**：按角色确定每个模块/页面的访问权限

**执行**：

1. 遍历 `function.actors[].permissions`，将权限映射到模块
2. 若角色有某模块中任一对象的操作权限，则该角色可访问该模块页面
3. 输出权限矩阵供用户确认

**permissions 为空时的降级策略**：

如果 `function.actors[].permissions` 为空（未在 D4 回填），从以下来源推断权限：
1. `function.scenarios[].actors[]` — 角色参与的场景 → 可访问场景涉及的模块页面
2. `function.processes[].steps[].actor` — 流程步骤的执行角色 → 可执行该步骤涉及的操作
3. 询问用户确认推断结果

**AI Agent 模块权限**：`is_ai_agent: true` 的模块不依赖 permissions 映射。权限来源为 D2 确认的"协作关系"——谁会使用这个 AI 功能，谁就能访问该页面。

**权限矩阵模板**：

```
页面权限矩阵：

| 角色 | {模块A页} | {模块B页} | 工作台 |
|------|----------|----------|--------|
| {角色1} | 可访问 | — | 可访问 |
| {角色2} | 可访问 | 可访问 | 可访问 |

需要调整哪些权限吗？
```

**对话示例**：


## R3：页面功能映射

**目标**：将场景和流程步骤映射到具体页面上的功能

**执行**：

1. 遍历 `function.scenarios[].workflow`，将每个 step 映射到对应模块页面的操作
2. 遍历 `function.processes[].steps`，将每个流程步骤映射到对应页面的操作按钮
3. 确保：流程中的每一步在原型中有对应的操作按钮或入口
4. 逐页面展示功能清单，逐一确认

**功能清单模板**：

```
{页面名} 功能清单：

| 来源 | 功能 | 操作类型 | 说明 |
|------|------|---------|------|
| 场景: {场景名} | {功能名} | 创建/查看/编辑/删除/业务操作 | {描述} |
| 流程: {流程名} Step{n} | {功能名} | 业务操作 | {流程步骤描述} |

```

## R4：页面流程设计

**目标**：定义页面之间的跳转关系和导航路径

**执行**：

1. 基于 `function.processes` 的步骤顺序定义页面跳转链
2. 基于状态转移（`transitions[]`）定义操作后的跳转
3. 如存在跨角色流程，规划工作台页面作为流程枢纽
4. 展示流程图（文本形式）

**流程图模板**：

```
页面跳转关系：

【{流程名}】
  工作台 → {模块A页}（操作：{触发操作}）
  {模块A页} → {模块B页}（操作：{触发操作}，条件：{状态条件}）
  {模块B页} → 工作台（操作：完成后返回）

【通用导航】
  侧边栏 → 各模块页（始终可访问）

这个跳转关系正确吗？
```

**对话示例**：

```
Agent：功能清单都确认好了。接下来咱们设计页面之间的跳转关系——就是用户做完一个操作后，会跳到哪个页面。

因为有一条跨角色的「订单履行」流程，我会设计一个工作台页面作为流程的中心。用户登录后先进入工作台，看到待处理的任务，点进去操作，完成后再回来。

页面跳转关系：

【订单履行】
  工作台 → 订单管理页（操作：点击"待确认订单"任务卡片）
  订单管理页 → 订单管理页（操作：确认订单后，订单状态变化，停留在当前页）
  工作台 → 订单管理页（操作：点击"待发货订单"任务卡片）
  订单管理页 → 工作台（操作：发货完成后返回工作台）
  工作台 → 订单管理页（操作：客户点击"待收货订单"任务卡片）
  订单管理页 → 工作台（操作：确认收货后返回工作台）

【通用导航】
  侧边栏 → 订单管理页（始终可访问）
  侧边栏 → 商品管理页（始终可访问）
  侧边栏 → 客户管理页（始终可访问）
  侧边栏 → 智能推荐页（始终可访问）
  顶部面包屑 → 任意上级页面（随时可返回）

这个跳转关系正确吗？

用户：工作台的设计思路我理解了。但是侧边栏导航和流程跳转会不会冲突？比如客户正在浏览商品，突然想去处理订单。

Agent：好问题。不会冲突的，我来解释一下设计思路：

页面上有两种导航方式并存：
• 侧边栏是"随时可用"的——不管用户在哪个页面、在做什么，都可以直接点侧边栏跳到任何模块页面
• 流程跳转是"引导式"的——工作台上的任务卡片引导用户按流程步骤操作，完成一步后自动回到工作台看下一步

这两种方式互不干扰：
  → 用户跟着流程走时，通过工作台任务卡片进入，操作完返回工作台
  → 用户想自由浏览时，随时点侧边栏，不受流程限制
  → 用户从侧边栏进入某个页面后，顶部面包屑可以随时返回上一级

所以您说的"正在浏览商品时想去处理订单"——直接点侧边栏的"订单管理"就行，不需要先回工作台。工作台上的任务只是提醒和引导，不是强制路径。

这样清楚了吗？

用户：清楚了，这样设计很合理。
```

**工作台设计**（仅当 `needs_workbench = true`）：
- 工作台作为所有流程的入口和进度追踪中心
- 从工作台可跳转到每个流程步骤对应的模块页面
- 完成操作后自动返回工作台查看进度

**完成条件**：用户确认跳转关系

## R5：页面结构设计

**目标**：为每个页面确定布局类型

**布局推荐规则**：

| 条件 | 推荐布局 | 典型场景 |
|------|---------|---------|
| 属性多（>8），数据结构化 | 表格列表 | 订单管理、客户列表 |
| 属性少（<=5），有图片/视觉元素 | 卡片视图 | 商品展示、员工名片 |
| 状态少（<=6个），强调流程推进 | 看板视图 | 任务管理、招聘流程 |
| 日期为核心维度 | 日历视图 | 会议预约、排班管理 |

**AI Agent 模块布局**（固定）：对话界面——顶部为 AI 对话区域（消息气泡 + 输入框），底部为对话历史。不适用对象/属性/状态维度判断。

**工作台布局**（固定）：任务看板 + 流程进度条 + 流程图

**评审工具布局**（固定，嵌入每个模块页面）：
- 折叠态：底部居中半透明黄色 Tab 页签（`rgba(250,204,21,0.85)`），向上突出约 28-32px
  - 显示"📋 评审 · {当前模块名}"，页面内容透过 Tab 可见
  - hover 时透明度提高到 0.95
- 展开态：底部全宽抽屉，高度 45vh，半透明深色底 + `backdrop-filter: blur(4px)` 毛玻璃效果
  - Tab 变为抽屉标题栏，保持视觉连续性（不消失再出现新元素）
- 非侵入性：`position: fixed; z-index: 1000`，不影响业务页面布局和交互
- 标签页：场景 | 流程 | 对象 | 操作 | 规则 | 权限矩阵（共 6 个）

**扩展机制**：当现有 4 种布局不适用时（如树形结构、地图视图、时间轴），Agent 可以：
1. 在推荐表中选择最接近的布局作为基础
2. 在对话中与用户讨论定制需求
3. 在页面结构方案中标注"定制布局"，说明特殊交互要求

## R6：页面交互设计

**目标**：确定每个页面的交互方式

**交互推荐规则**：

| 维度 | 条件 | 推荐方式 |
|------|------|---------|
| 详情查看 | 属性多/有编辑需求 | 模态框 |
| 详情查看 | 只读查看 | 侧边抽屉 |
| 新建方式 | 属性少（<=6） | 模态框 |
| 新建方式 | 属性多或分步 | 独立页面/分步向导 |
| 操作按钮 | 操作少（<=3） | 行内按钮 |
| 操作按钮 | 操作多（>3） | 顶部工具栏 |
| 关联数据 | 1:1 关系 | 行内展示 |
| 关联数据 | 1:N 关系 | 标签页子表格 |

**工作台交互**（固定）：点击任务卡片跳转模块页 + 任务卡片操作按钮 + 流程进度条

**评审工具交互**（固定，嵌入每个模块页面）：
- 折叠态交互：点击半透明 Tab 展开抽屉
- 展开态交互：标签页切换，各标签页展示当前模块的 FBS 模型数据
- 上下文联动：切换页面时，Tab 文字和抽屉内容自动更新为当前模块数据
  - 场景标签页：标注与当前页面相关的 workflow 步骤
  - 流程标签页：高亮当前模块对应的流程步骤
  - 权限矩阵标签页：高亮当前角色行和当前模块列

**AI Agent 模块交互**（固定）：
- 页面主体为 AI 对话界面：消息气泡 + 输入框 + 发送按钮
- 其他页面中的入口：固定为右下角悬浮按钮（点击打开 AI 对话弹窗或跳转 AI 页面）
- 无表格、无列表、无新建/编辑表单

## 映射表与参考数据

以下表格供 Agent 在 R1–R6 各步骤中查阅，确保映射一致性。

### 1. 模型到界面映射

| 模型元素 | YAML 路径 | 界面元素 |
|---------|----------|---------|
| 对象 | `domain.objects[]` | 管理页面（一个模块一个页面） |
| 属性 | `objects[].attributes[]` | 表单字段 / 列表列 / 详情项 |
| 枚举属性 | `attributes[type=enum]` | 下拉选择 / 状态标签（带颜色） |
| 关联属性 | `attributes[type=reference]` | 选择器弹窗 / 自动补全 |
| 关系 1:1 | `relationships[cardinality=one_to_one]` | 详情页行内展示 |
| 关系 1:N | `relationships[cardinality=one_to_many]` | 标签页子表格 |
| 状态 | `objects[].states[]` | 状态标签（带颜色和图标） |
| 转移 | `objects[].transitions[]` | 操作按钮（按状态动态显示） |
| 操作 | `objects[].operations[]` | 页面按钮（位置由交互设计决定） |
| 业务规则 | `objects[].rules[]` | 表单校验提示 / 操作确认弹窗 |
| 角色 | `function.actors[]` | 页面访问权限 / 侧边栏可见性 |
| 场景 | `function.scenarios[]` | 页面功能集合 |
| 流程 | `function.processes[]` | 工作台流程图 / 进度追踪 |

### 2. 状态机映射模板

对每个有状态的对象，按此模板生成状态机数据：

```yaml
# 对象: {对象名}
state_machine:
  initial_state: "{is_initial=true 的状态名}"
  states:
    - name: "{状态名}"
      label: "{中文名}"
      color: "{UI色值}"    # 从 data_dictionary 或自动分配
      icon: "{图标名}"     # 可选
      is_final: {true/false}
  transitions:
    - from: "{源状态}"
      to: "{目标状态}"
      trigger: "{操作名}"
      trigger_label: "{操作中文名}"
      condition: "{附加条件，可选}"
```

### 3. 业务规则索引模板

按对象索引所有规则，供原型生成时实现校验逻辑：

```yaml
# 对象: {对象名}
business_rules:
  object_rules:        # 对象级规则（不变量）
    - id: "{rule_id}"
      constraint: "{约束表达式}"
      message: "{错误提示}"
      severity: "{error/warning}"
  operation_rules:     # 操作级规则（前置条件）
    - operation: "{操作名}"
      rules:
        - condition: "{条件}"
          action: "{ALLOW/DENY/WARN}"
          message: "{提示信息}"
```

### 4. 原型页面参考

| 原型文件 | 对应模型 | 生成条件 |
|---------|---------|---------|
| `prototype/index.html` | 全局导航 + 所有模块入口 | 必须生成 |
| `prototype/workbench.html` | `function.processes[]` | 仅当 processes 非空 |
| `prototype/{module}.html` | `domain.modules[]` + 所属 objects | 每个模块一个页面 |
| `prototype/{module}-form.html` | `objects[].attributes[]` | 仅当新建方式为"独立页面" |
| `prototype/styles.css` | 全局样式 | 必须生成 |
| `prototype/app.js` | 模拟数据 + 交互逻辑 + 流程模拟 | 必须生成 |

### 5. 建议实现顺序

基于 `domain.modules[].depends_on` 依赖关系，按以下规则确定页面生成顺序：

1. **拓扑排序**：无依赖的模块先生成，被依赖的模块先生成
2. **无依赖关系时**：按对象数量从少到多排列
3. **工作台**：始终在最后生成（依赖所有模块页面的路由）

```
生成顺序示例：
1. {模块A}（无依赖，对象数少）
2. {模块B}（依赖模块A）
3. {模块C}（无依赖，对象数多）
4. 工作台（依赖所有模块页面）
5. index.html（依赖所有页面路由）
```
