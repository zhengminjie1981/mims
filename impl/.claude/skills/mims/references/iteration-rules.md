# 设计迭代

> **适用范围**：设计过程中及原型后的统一变更管理。无论是否已有原型，均适用本流程。

---

## 触发条件

以下方式均可触发：

**设计过程中**：用户说"修改"/"改一下"/"不对"/"还有"等词语时。

**原型后反馈**：用户基于原型体验提出修改意见时（反馈收集见 `workflow-prototype.md` R9）。当存在 `prototype/feedback.md` 且非空时，优先读取该文件作为反馈来源，无需用户手动复述。读取后向用户展示提取的反馈列表并确认。

**Agent 检测触发**：建模过程中发现以下问题：
- 发现新对象与已有对象存在未声明的关系
- 检测到规则冲突（两条操作规则互相矛盾）
- 发现引用断裂（某操作/场景引用了不存在的对象或状态）
- 用户描述与已建模内容矛盾

Agent 检测触发时，先向用户说明问题，再询问是否进入变更流程。

---

## 反馈分类（原型后触发时）

原型后反馈需先分类，设计过程中的变更跳过此步直接进入变更级别识别。

| 类型 | 特征 | 例子 |
|------|------|------|
| **简单反馈** | 不涉及业务逻辑变更，只影响 UI 表现 | 改标签文案、调整布局位置、改颜色/图标、字段显示顺序、按钮位置 |
| **复杂反馈** | 涉及业务逻辑、数据结构、流程或角色的变更 | 审批流程不对、缺少功能页面、角色权限不合理、业务规则需调整 |

**判断原则**：反馈内容如果需要修改 `domain-model.yaml` 的 F 层（场景/流程）或 S/B 层（对象/状态/操作/规则），则为复杂反馈。

**简单反馈处理**：直接修改原型文件 + 检查是否需同步模型（如标签文案对应 `chinese_name`）。不走设计迭代流程。

---

## 策划（复杂反馈时执行）

复杂反馈涉及业务逻辑变更，需先策划再执行。

### 1. 整理反馈

将用户的所有反馈整理为结构化列表：

```
反馈整理：

| # | 反馈内容 | 涉及要素 | 影响判断 |
|---|---------|---------|---------|
| 1 | 审批流程需要加一级 | 场景、流程、状态、操作 | 复杂 |
| 2 | 列表里少了一个筛选条件 | 属性 | 简单，可一并处理 |
```

### 2. 分析影响

基于反馈内容，分析对设计的影响范围：
- 受影响的场景/流程（F 层）
- 受影响的对象/属性/关系（S 层）
- 受影响的状态/操作/规则（B 层）
- 受影响的原型页面（如有）

### 3. 制定变更计划

明确变更步骤并请用户确认：

```
变更计划：

设计迭代：
  ├─ 回到 P5（流程步骤补充一级审批）
  ├─ 回到 D3（状态机调整：新增审批状态）
  ├─ 回到 D4（操作和规则调整）
  └─ 验证 + 更新 srs.md / sdd.md

原型调整（设计迭代完成后自动执行）：
  ├─ 受影响页面：订单管理页、工作台
  ├─ 调整方式：增量修改
  └─ 预估调整范围：{具体说明}

请确认这个计划是否合适？
```

### 4. 用户确认并记录

用户确认后，将变更计划写入 `sdd.md` 变更记录，然后进入设计迭代执行。

**设计过程中触发的变更**：如触发时已有策划阶段已明确的计划，直接按计划执行，不再重新识别级别和展示影响。

---

## 变更级别与处理

| 级别 | 判断标准 | 处理动作 |
|------|---------|---------|
| **L1 轻微** | 只改名称、描述、标签、中文名 | 当前步骤直接修改，无需回退 |
| **L2 局部** | 改单个对象的属性/关系/操作/规则 | 回退到该对象所在的建模步骤 |
| **L3 中等** | 新增对象，或多对象协调修改 | 回退到所属阶段起点（初步设计→P3，详细设计→D1），保留已有内容供参考 |
| **L4 重大** | 删除对象，或改变整体架构/核心场景 | 展示完整影响分析，提供"新建项目"或"当前重建"两个选项 |

---

## 执行流程

1. **识别级别** — 判断 L1/L2/L3/L4，向用户说明（策划阶段已明确的跳过此步）
2. **变更前验证** — 检查以下问题，若存在则一并告知用户：
   - 变更后是否产生孤立对象（无任何关系的对象）
   - 是否有关系因对象删除而断裂
   - 是否有操作/规则引用了将被删除的对象或状态
   - 是否有场景因对象/操作变更而无法执行
3. **展示影响** — 列出受影响的对象、关系、操作、场景、文档、原型
4. **获取确认** — L3/L4 必须用户明确确认后才执行；L1/L2 可直接执行
5. **保存快照** — 执行前记录 `metadata.last_updated` 时间戳（用于回滚参照）
6. **执行修改** — 按回退动作修改模型
7. **重新验证** — 委托 `mims-validator` 验证修改后的模型（根据当前阶段选择 mode）
8. **同步文档** — 变更涉及初步设计内容时更新 `srs.md`；涉及详细设计内容时更新 `sdd.md`
9. **记录变更文档** — 将实际变更内容写入变更记录（见下方"变更记录格式"），作为原型调整的输入依据
10. **原型相关处理** — 根据原型阶段状态自动处理（见下方"原型相关处理"章节）

---

## 展示影响

展示影响时，逐条列出受影响的：
- 对象（新增/删除/修改）
- 关系（需要调整的）
- 操作和规则（失效或需更新的）
- 场景（需要更新的）
- 文档（srs.md / sdd.md 是否需要更新）
- 原型（受影响的页面，如有）

---

## 原型相关处理

> 设计迭代执行完成后（步骤 9），根据原型阶段的状态自动处理。不同于首次原型生成（R7 委托子代理全量生成），此处基于**变更记录**中的实际变更内容进行定向调整。

### 判断原型阶段状态

执行完设计迭代后，检查原型阶段状态：

| 状态 | 判断依据 | 处理 |
|------|---------|------|
| **原型未开始** | `metadata.design_progress.prototype.completed_steps` 为空 | 不自动进入原型阶段。用户按正常流程进入即可。 |
| **原型规划已完成（R1-R6），未生成** | `prototype/` 目录不存在，但 `design_progress.prototype.completed_steps` 包含 R1-R6 中的步骤 | 标记受影响的规划步骤需重新审视，后续进入原型阶段时从最早受影响的步骤重新执行 |
| **原型已生成（R7+）** | `prototype/` 目录存在 | 基于变更记录执行原型调整（见下方"原型调整"章节） |

### 原型规划需重新审视的标记

当原型规划已完成但未生成时，对照变更记录判断哪些规划步骤受影响：

| 变更涉及 | 需重新审视的规划步骤 |
|---------|-------------------|
| 角色/权限变更 | R2（页面权限规划） |
| 场景/操作变更 | R3（页面功能映射） |
| 流程变更 | R4（页面流程设计） |
| 对象/属性结构变更 | R5（页面结构设计） |
| 状态/操作/规则变更 | R6（页面交互设计） |

标记方式：将受影响的步骤从 `design_progress.prototype.completed_steps` 中移除。用户后续进入原型阶段时，自动从最早的未完成步骤开始。

### 原型调整

**前提条件**：`prototype/` 目录存在，且有步骤 9 产出的变更记录。

#### 评估原型影响

对照变更记录中的实际变更内容，判断受影响的页面：

| 变更类型 | 对原型的影响 |
|---------|------------|
| 新增/删除属性 | 列表列、表单字段、详情项 |
| 新增/删除状态 | 状态标签、操作按钮、状态机数据 |
| 新增/删除操作 | 页面按钮、确认弹窗 |
| 新增/删除对象 | 新增/移除管理页面 |
| 流程变更 | 工作台流程图 |
| 角色权限变更 | 侧边栏可见性、页面访问权限 |
| 业务规则变更 | 表单校验、操作确认逻辑 |

#### 确定调整方式

| 调整规模 | 判断标准 | 处理方式 |
|---------|---------|---------|
| **增量调整** | 变更影响 ≤3 个页面，且页面结构不变 | Agent 直接修改受影响的 HTML 文件 |
| **部分重新生成** | 变更影响 >3 个页面，或页面结构有变化 | 提示用户确认后，重新生成受影响的页面 |
| **全量重新生成** | 核心场景/角色变化，或大部分页面受影响 | 提示用户确认后，委托 mims-prototyper 全量重新生成 |

#### 执行与验证

1. 按确定的调整方式修改原型文件
2. 对照变更记录逐项确认：每项变更在原型中是否已体现
3. 检查调整后的原型与更新后的模型一致性
4. 向用户展示调整结果

#### 与首次原型生成的区别

| 维度 | 首次生成（R7） | 原型调整（迭代中） |
|------|-------------|----------------|
| 输入 | 完整的 domain-model.yaml | 变更记录 + 变更后的 model |
| 范围 | 全部页面 | 仅受影响的页面 |
| 执行者 | mims-prototyper 子代理 | Agent 直接修改（全量时除外） |
| 验证 | R8 端到端流程验证 | 对照变更记录逐项确认 |

---

## 快照时机

| 时机 | 动作 |
|------|------|
| 每个建模步骤完成时 | 将当前 `domain-model.yaml` 写入磁盘（覆盖更新） |
| 变更执行前（步骤 5） | 记录 metadata.last_updated 时间戳 |
| 检查点通过时 | 更新 srs.md 或 sdd.md |

## 变更记录格式

> 变更记录是执行流程步骤 9 的产出，追加到 `sdd.md` 关键决策记录章节。
> 它同时是原型调整的**输入依据**——原型调整基于变更记录中的实际变更内容进行影响分析和定向调整。

**设计过程中触发**：

```markdown
## 变更记录 #{n} — {datetime}

- 变更级别：L{n}（{轻微/局部/中等/重大}）
- 触发方式：{用户主动 / Agent检测}
- 变更内容：
  - {对象A：{具体修改了什么}}
  - {对象B：{具体修改了什么}}
  - {场景/流程：{具体修改了什么}}
- 影响范围：{受影响的对象/关系/操作/场景列表}
- 原型状态：{未开始 / 规划已完成 / 已生成}
- 原型处理：{无需处理 / 标记重新审视 R{n}/R{n} / 增量调整{页面列表} / 重新生成}
- 用户确认：是
```

**原型后触发**：

```markdown
## 原型迭代 #{n} — {datetime}

- 反馈来源：{基于原型操作体验 / 基于原型截图标注}
- 反馈类型：{简单反馈 / 复杂反馈}
- 反馈内容：{逐条列出}
- 变更计划：{复杂反馈时，摘要说明}
- 执行结果：
  - {设计变更：{对象A 修改了什么}}
  - {设计变更：{对象B 修改了什么}}
- 原型处理：{简单直接修改 / 增量调整{页面列表} / 部分重新生成{页面列表} / 全量重新生成}
- 用户确认：是
```
