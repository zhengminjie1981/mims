# 初步设计步骤 P1–P6 详细执行指南

> 本文档描述初步设计阶段每个步骤的具体执行方式，由 SKILL.md 在进入对应步骤时引用。
>
> **覆盖范围**：P1（需求收集）→ P2（资料理解）→ P3（背景与目标）→ P4（角色与场景）→ P5（业务流程）→ P6（架构总览与检查点）
>
> **设计阶段标记**：进入 P1 时，写入 `metadata.design_phase = "preliminary"`
>
> **YAML 结构参考**：所有字段的完整 Schema 定义见 `schema.md`，本文档仅描述写入要点。

---

## P1：需求收集 + 准备引导

**目标**：获取初步需求，引导用户准备材料，写入 `metadata` 基本信息。

**执行流程**：

1. **准备引导**（仅首次，`domain-model.yaml` 不存在或进度为空时）：

> "在开始之前，如果您有以下任何材料，建议先准备好再告诉我：
>  • 对系统的大致构想或目标
>  • 目标用户是谁，他们做什么工作
>  • 主要的业务流程或工作步骤
>  • 现有系统的截图、文档、表格样板
>  • 任何您已经整理过的资料
>  准备得越充分，对话效率越高。当然，只有模糊的想法也可以直接开始。"

2. **开放式提问**："请告诉我，您想做一个什么样的系统？主要用来解决什么问题？"

3. **适用性评估**（静默判断，不单独提问）：

   根据用户描述，通过关键词匹配和语义理解，判断系统类型和适用程度。评估结果用于步骤 4 的反馈语气。

   **软件类型五级分类**：

   | 等级 | 覆盖度 | 类型 | 核心价值在于 |
   |------|--------|------|------------|
   | 高度适用 | ≥ 85% | 管理信息化（ERP/CRM/OA/订单/库存/采购）、电商交易（B2C/B2B/支付/预订） | 业务流程 + 状态机 + 规则约束 |
   | 较高适用 | 70-84% | 内容管理（CMS/素材库）、教育培训（LMS/考试）、生产力工具（项目管理/笔记） | 有业务对象和流程，部分技术细节超出模型 |
   | 中度适用 | 50-69% | IoT 管理（智能家居/设备管理）、通信社交（IM/协同办公） | 业务层可建模，核心价值在实时/硬件层 |
   | 低度适用 | 30-49% | 前端可视化（BI/大屏/低代码）、游戏娱乐、平台基建 | 仅能建模外围管理功能 |
   | 不适用 | < 30% | 算法密集（推荐/搜索/ML）、嵌入式驱动、科学计算（CAD/仿真） | 核心价值与业务建模无关 |

   **关键词映射**：

   高度适用词（出现 ≥ 2 个 → 判为"高度适用"）：
   > 管理、流程、审批、单据、台账、档案、角色、权限、状态、流转、库存、订单、客户、供应商、报销、出入库、审核、归档、电商、采购、销售、发货、退货、支付、预订、合同

   关注词（出现 → 检查是否混合需求）：
   > 报表、统计、看板、消息推送、消息通知、第三方对接、设备管理、IoT、传感器、智能家居、直播、视频会议、即时通讯、群聊、笔记、文档协作、课程、考试

   不适用词（出现 ≥ 2 个 → 判为"不适用"）：
   > 智能推荐、自动预测、实时监控、算法、机器学习、深度学习、最优、毫秒级、流式处理、实时计算、大数据分析、图像识别、语音识别、自然语言处理、搜索引擎、嵌入式、单片机、驱动开发、固件、实时操作系统、CAD、仿真、有限元、数值计算、物理模拟

   **判断规则**：
   - 高度适用词 ≥ 2 且无不适用词 → **第一档：高度适用**
   - 高度适用词 ≥ 1 且有不适用词 ≥ 1 → **第二档：部分适用**
   - 不适用词 ≥ 2 且无高度适用词 → **第三档：不适用**
   - 仅有关注词 → 根据语义判断，倾向第二档
   - 无法判断 → 默认第一档，在后续步骤中持续观察

4. **复述确认 + 适用性提示**：

   在复述理解的同时，根据步骤 3 的评估结果给出明确提示。**必须**给出适用性提示，不可省略。

   **第一档（高度适用）** — 一句话带过：

   > "我理解您想做的是：{复述}。这是一个{管理/交易/内容管理}类的系统，正好是我最擅长的方向。对吗？"

   **第二档（部分适用）** — 拆解说明：

   > "我理解您想做的是：{复述}。
   >
   > 这个系统有两部分需求：
   > - **{管理/流程/订单}部分**：这部分我比较擅长，可以帮您梳理清楚角色、流程和规则。
   > - **{数据分析/实时通信/AI推荐}部分**：这部分超出我的建模能力，建议最终交给专业的{数据工程师/AI工程师}来落地。我可以帮您把这部分的需求意图记录下来。
   >
   > 咱们先从{管理部分}开始，您觉得呢？"

   **第三档（不适用）** — 诚实告知 + 引导：

   > "我理解您想做的是：{复述}。
   >
   > 坦诚地说，这个系统的核心是{算法推荐/实时计算/科学仿真}，这不是我擅长的领域。我最擅长的是帮您设计**业务管理系统**——比如管理订单、审批流程、库存台账这类。
   >
   > 不过，如果您这个系统中有**管理方面的需求**（比如用户管理、权限控制、配置管理），我可以帮您先把这部分梳理出来。如果没有的话，可能需要找专业的{算法团队/嵌入式工程师}来帮您。
   >
   > 您看怎么安排？"

   用户确认或调整后 → 追问："目前是怎么处理的？哪里最麻烦？"

5. **询问资料**："您有没有需求文档、产品说明书、截图或其他参考资料？A. 有文档或文件  B. 有网页链接  C. 没有，我直接说"
   - A/B → 进入 P2
   - C → 跳过 P2，进入 P3

6. **写入** `metadata.project_name`、`metadata.description`；提取业务领域、核心目的、痛点写入 `metadata.context`；标记 `design_progress.preliminary.status = "in_progress"`；若为第二档或第三档，将适用性评估结论写入 `metadata.context.suitability_note`。

**完成条件**：用户确认系统基本定位；适用性评估已完成并反馈用户；project_name 和 description 已写入。

## P2：资料理解（可选）

**触发**：P1 中用户提供了参考资料。

**目标**：读取资料 → 提取关键信息 → 生成摘要 → 用户确认 → 写入 `source_materials`。

**执行流程**：

1. **判断资料类型并读取**：document（本地文件）/ image（图片识别）/ url（网页抓取）/ text（直接使用）。

2. **提取 13 类关键信息**（类别见 `schema.md` §3.6.4）：business_goal、user_role、business_object、attribute、relationship、business_rule、operation、constraint、scenario、pain_point、requirement、assumption、other。
   - 忠实原文，不臆测
   - 每条评估置信度（0–1），低于 0.70 标注"待确认"
   - 记录来源位置（页码/段落/区域）

3. **生成摘要**展示给用户：

> "我已经阅读了您提供的《{资料名称}》，提取了以下关键信息：
>  【业务目标】{goals}
>  【目标用户】{roles}
>  【核心对象】{objects}
>  【业务规则】{rules}
>  【使用场景】{scenarios}
>  这些信息准确吗？有没有需要补充或修改的地方？"

4. **处理反馈**：确认→继续；纠正→更新并降低置信度；补充→添加新信息。

5. **写入** `source_materials`（结构见 `schema.md` §3.6），确保 `processing.status = "completed"`。

**后续引用**：P3–P6 中统一用 "根据您提供的《{资料名称}》，我了解到：{内容}"。

**完成条件**：资料已读取、摘要已确认、source_materials 已写入。

## P3：背景与目标确认

**目标**：确认业务背景、使用环境、成功标准，补全 `metadata.context`。

**执行流程**：

1. "这是公司内部使用的系统，还是面向外部客户的？" + "大概有多少人会同时使用？"
2. "有没有什么特殊的限制条件？比如必须使用特定设备、需要离线使用、有合规要求等。"
3. "如果这个系统做成功了，您觉得最重要的是达成什么目标？"
4. **资料交叉**：如果 P2 提取了 business_goal 或 constraint，主动引用："根据您提供的《{资料名}》，我了解到目标是 {goal}。结合您刚才说的……对吗？"
5. **写入** `metadata.context`：industry、scale、usage_type、special_constraints、success_criteria。

**对话示例**：

```
Agent：我们来确认一下使用背景。这个系统是公司内部用的，还是面向外部客户的？

用户：主要是公司内部用，仓库管理员和采购员用的。

Agent：明白，企业内部系统。大概有多少人会同时使用？比如十几个人？还是上百人？

用户：不多，大概 5 到 10 个人同时用吧。

Agent：好的，中等规模。有没有什么特殊的要求？
  比如必须用手机操作、需要离线使用、或者有什么行业合规要求？

用户：仓库那边有时候信号不太好，希望能支持手机操作。其他没什么特别的。

Agent：我确认一下：
  • 使用类型：企业内部
  • 使用规模：约 5-10 人同时在线
  • 特殊约束：需要支持手机端操作
  对吗？还有一个问题——如果这个系统做成功了，
  您觉得最重要的是达成什么目标？

用户：最主要是减少库存差异，现在人工记账经常对不上。

Agent：了解，核心目标是"减少库存差异，提高账实准确率"。
我把这些都记录下来了。
```

**完成条件**：使用对象和环境已明确；metadata.context 已补全。

## P4：用户角色与业务场景（F 功能层）

**目标**：识别 `function.actors[]` 和 `function.scenarios[]`。

**关键**：场景的 `actors` 是数组——一个场景可涉及多个角色，一个角色参与多个场景（多对多）。

**执行流程**：

1. **识别角色**：
   - 有资料提取（`category == "user_role"`）："根据您提供的《{资料名}》，我识别出这些使用者：{列表}。对吗？还有遗漏的吗？"
   - 无资料："谁会使用这个系统？请告诉我都有哪些不同身份的人。"

2. **确认角色职责**（每角色一问）："【{角色名}】的主要工作是什么？他们在这个系统里需要做什么？"

3. **识别场景**（对每个角色）："对于【{角色名}】，能描述一个最常见的工作场景吗？从开始到结束大概是怎么做的？"

4. **完善场景要素**："这个场景开始之前需要满足什么条件？做完之后会产生什么结果？"

5. **多角色场景**：如涉及协作，确认各自职责并记录所有参与角色到 `actors` 数组。

6. **写入** `function.actors[]` 和 `function.scenarios[]`（含 workflow）。

**小结格式**：角色表（职责 + 参与场景）+ 场景表（参与角色 + 前提 + 结果）。

**对话示例**：

```
Agent：现在来梳理谁会用这个系统。根据您之前的描述，我理解有两类使用者：
  1. 仓库管理员 — 负责日常出入库和盘点
  2. 采购员 — 负责下采购订单
  对吗？还有其他角色吗？

用户：对，还有经理要看报表。

Agent：好的，一共 3 个角色。我先确认一下「仓库管理员」：
  他在这个系统里主要做什么操作？

用户：扫码入库、扫码出库、定期盘点。

Agent：那「采购员」呢？主要做什么？

用户：看库存不够了就下采购单，收到货之后确认入库。

Agent：我注意到「确认入库」这件事——采购员确认之后，
  是不是还需要仓库管理员去做实际的扫码入库？

用户：对，采购员在系统里确认到货，然后仓库那边实际扫码入库。

Agent：明白，这是一个需要两个人配合的场景。我来整理一下：

  场景一：采购到货入库
    • 参与角色：采购员、仓库管理员
    • 前提：采购单已审批通过，货物已送到
    • 结果：货物入库完成，库存数量更新

  场景二：日常盘点
    • 参与角色：仓库管理员
    • 前提：到达盘点周期
    • 结果：生成盘点差异报告

  还有遗漏的场景吗？

用户：还有库存预警，库存低于最低值的时候要通知采购员。

Agent：好的，补上第三个：

  场景三：库存预警与补货
    • 参与角色：系统（自动触发）、采购员
    • 前提：某商品库存低于安全值
    • 结果：采购员收到通知并创建采购单

  这三个场景覆盖完整了吗？
```

**完成条件**：每角色 ≥1 个场景；每场景有前/后置条件；actors 和 scenarios 已写入。

## P5：业务流程梳理（F 功能层）

**目标**：识别跨角色端到端流程，写入 `function.processes[]`。

**场景与流程的关系**：场景的 workflow 是概述性描述（"大概有哪些步骤"），流程的 steps 是对场景的细化展开——包含每步的 actor、module、info_passed、business_logic 等详细信息。

**关键规则**：每个流程必须有 `parent_scenario`（引用场景 id）。单角色系统无跨角色协作时跳过此步。

**执行流程**：

1. **判断是否需要**："在这些场景中，有没有需要多个角色配合才能完成的完整流程？" → 用户确认不存在 → 跳过 P5 进入 P6。

2. **逐个梳理**："请描述一下「{流程名}」从头到尾是怎么走完的？每一步由谁来做？"

3. **收集要素**：归属场景（parent_scenario）、参与角色（从步骤推断）、起止条件。

4. **细化每个步骤**，收集：actor（谁来做）、module（涉及哪个功能区域，自由文本描述，如"订单相关"、"客户相关"，P6 会统一确认模块名）、info_passed（传递什么信息，从谁到谁）、business_logic（顶层业务逻辑）。

5. **写入** `function.processes[]`（步骤含 module、info_passed、business_logic 字段，结构见 `schema.md` §3.4.3）。

**小结格式**：每条流程展示 起步→步骤（角色→操作→模块→信息流）→完成。

**对话示例**：

```
Agent：在您提到的场景中，「采购到货入库」涉及采购员和仓库管理员两个人配合。
  能不能从头到尾描述一下这个流程是怎么走完的？

用户：采购员先下采购单，经理审批之后发给供应商。
  货到了之后采购员在系统里确认收货，然后仓库管理员去扫码入库。

Agent：我梳理一下这个流程，看看对不对：

  流程：采购到货入库（属于场景「采购到货入库」）

  步骤 1：采购员 → 创建采购单 → 采购管理模块
    传递：采购员将商品名称、数量、供应商信息录入系统

  步骤 2：经理 → 审批采购单 → 采购管理模块
    传递：审批意见从经理到采购员

  步骤 3：采购员 → 确认收货 → 采购管理模块
    传递：到货确认信息

  步骤 4：仓库管理员 → 扫码入库 → 库存管理模块
    传递：实际入库数量从仓库管理员到库存记录

  每一步都对应上了吗？有没有哪里不对？

用户：差不多，不过步骤 2 审批如果不通过的话要退回给采购员修改。

Agent：好的，补充一个分支逻辑：
  步骤 2 审批不通过 → 采购员修改采购单 → 重新提交审批。
  那这个流程就形成了一个循环，直到审批通过才算继续往下走。

  还有其他需要补充的吗？比如有没有超时处理、异常情况？

用户：没有了，就这样。

Agent：好的，这个流程完整了。还有「库存预警与补货」的流程需要梳理吗？

用户：那个比较简单，系统检测到库存低了就通知采购员，
  采购员直接下采购单就行。

Agent：明白，我整理一下：

  流程：库存预警与补货（属于场景「库存预警与补货」）

  步骤 1：系统自动 → 检测库存低于安全值 → 库存管理模块
    业务逻辑：比对当前库存与安全库存阈值

  步骤 2：系统自动 → 发送预警通知 → 通知模块
    传递：预警信息从系统到采购员

  步骤 3：采购员 → 创建采购单 → 采购管理模块
    传递：补货商品信息从预警通知到采购单

  这两个流程就是全部了吗？
```

**完成条件**：每流程有 parent_scenario；每步骤有 actor 和 module；processes 已写入。

## P6：架构总览与检查点

**目标**：综合 F 层成果，推断模块边界和外部接口，评估 AI 适用性，执行验证并生成 SRS。

### 6.1 综合小结

> "我们已经梳理清楚了系统的整体脉络：
>  【使用者】{n} 个角色 | 【使用场景】{n} 个 | 【业务流程】{n} 个
>  接下来，我来梳理系统的整体架构和模块划分。"

### 6.2 推断模块边界

从场景和流程归纳模块（参考 P5 步骤的 module 字段），展示模块划分表（模块名 + 包含内容 + 主要使用者），请用户确认。

**功能-结构背离检测**：推断模块时同步检查以下信号，若出现则主动引导：

| 信号 | 含义 | 引导方式 |
|------|------|---------|
| 模块内对象 > 6 个且缺少直接关联 | 功能区 ≠ 结构域 | 建议拆分为更小的、内聚的子模块 |
| 用户描述的"模块"本质是端到端流程 | 混淆了功能区和结构域 | 建议用「场景」或「业务流程」表达跨模块协作，模块只保留核心对象 |
| 单个场景涉及 3+ 个模块的对象 | 跨域协作不应归入同一模块 | 确认模块划分，将协作逻辑留在场景/流程中 |

**引导话术示例**：
```
我注意到【{模块名}】包含了{多}个关联不太紧密的对象，比如{A}和{B}。
这两类东西业务上差别比较大，建议分成两个独立的功能模块。
它们之间的协作可以通过场景和流程来描述，不影响整体使用。

您觉得是分开好，还是先放一起？
```

### 6.3 识别外部系统交互

询问："有没有需要跟外部系统打交道的地方？比如支付、发短信、对接 ERP、调地图接口等。"
如有，逐个确认 direction、trigger、request/response_data，写入 `external_interfaces`。

### 6.4 AI Agent 评估（按需）

当用户提到某模块需要"智能""自动""AI"时，执行 6 维度评估：

| 维度 | 评估内容 |
|------|---------|
| 输入特征 | 是否非结构化（自然语言、图片等） |
| 处理逻辑 | 是否需要判断、推理或创造性处理 |
| 输出特征 | 是否开放性（生成内容、建议等） |
| 精度要求 | 是否允许概率性结果 |
| 变化频率 | 规则是否频繁变化 |
| 学习需求 | 是否需要从数据中学习改进 |

展示评估表 + 建议（AI Agent / 传统 / 混合）+ 理由。确认后立即写入 `domain.modules[]` 对应模块的 `is_ai_agent: true`、`ai_agent_type`、`ai_justification` 字段。不等到后续步骤。

### 6.5 检查点仪式

1. **委托验证**：`mims-validator`（mode: preliminary），重点检查 F 层完整性和一致性。
2. **置信度决策**：≥50% → 继续；<50% → 列出缺口，引导补充后重验。
3. **验证未通过时的引导**：
   - 置信度 50-70%：列出具体缺口，用自然语言逐项请用户补充。"我检查了一下，发现还需要补充以下内容：{逐条列出}。我们先从第一个开始？"
   - 置信度 <50%：列出必须解决的问题，引导用户逐个解决。"要继续往下走，有几个关键问题需要先解决：{逐条列出}。"
   - 每次补充后重新验证，不跳过未解决的问题。
4. **生成 SRS**：置信度 ≥50% 时，委托 mims-spec-generator（type: srs）。委托前先加载 `references/srs-template.md`，将模板内容一并传入。
5. **展示摘要**：角色数、场景数、流程数、模块数、外部接口数、置信度、已生成文件。
6. **用户确认** → 更新 `design_progress.preliminary.status = "complete"` → 展示阶段过渡引导（见 `workflow-common.md` §五 P6→D1）→ 进入详细设计（D1）。
