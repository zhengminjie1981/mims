## 1. 这是什么

### 迷悟师能做什么

您有一个想法——比如"我想做一个员工报销管理系统"——但不知道从哪里下手？

迷悟师会像顾问一样，通过一系列问题，帮您把这个模糊的想法变成：

- **一份清晰的业务设计**（记录在 `domain-model.yaml` 文件中）
- **一个可以点击演示的原型**（直接在浏览器打开，展示给同事或开发团队）

整个过程不需要您懂任何技术，只需要用日常语言描述您的业务需求。

### 适合哪些人使用

| 适合 | 典型用途 |
|------|---------|
| 产品经理 | 在和开发团队沟通前，把需求整理清楚 |
| 业务部门负责人 | 描述业务流程，提出系统化管理需求 |
| 创业者 | 验证产品想法，制作演示原型 |
| 业务分析师 | 快速建立业务模型，输出结构化需求文档 |

### 不适合哪些场景

- 需要搭建真正可上线运行的系统（迷悟师只做**原型**，不是生产代码）
- 已有明确技术方案，只需要编码实现
- 纯技术问题（如服务器配置、数据库优化）

## 2. 安装配置

### 前提条件

您需要先安装 **Claude Code**（命令行工具）。

安装方法：参考 [Claude Code 安装指南](https://gitlab.xyitech.com/Antbook/resources/blob/master/ClaudeCode-Obsidian-%E5%AE%89%E8%A3%85%E6%8C%87%E5%8D%97.md)，安装完成后在终端输入 `claude --version` 能看到版本号即为成功。

### 三步安装迷悟师

**第一步**：将迷悟师的文件复制到您的项目目录

假设您的项目目录是 `/Users/wang/my-project`：

```bash
cp -r impl/. /Users/wang/my-project/
```

复制后，您的项目目录结构如下：

```
my-project/
├── CLAUDE.md                    ← 迷悟师的"说明书"（不要删除）
└── .claude/
    ├── agents/                  ← 迷悟师的两个助手（不要修改）
    └── skills/mims/             ← 迷悟师的工作流程（不要修改）
```

> **提示**：`.claude` 目录是隐藏目录，如果您的文件管理器看不到，这是正常的。

**第二步**：进入您的项目目录，启动 Claude Code

```bash
cd /Users/wang/my-project
claude
```

**第三步**：输入以下命令，启动迷悟师

```
/mims
```

看到迷悟师的开场白，即安装成功。

## 3. 第一次使用：完整流程示例

为了让您快速上手，这里以"**餐厅点餐管理系统**"为例，演示完整的使用流程。

### 3.1 启动

输入 `/mims design` 后，迷悟师会说：

```
您好！我是迷悟师，帮您把想法变得清晰、有条理。

我们会分两个阶段完成：
1. 需求建模：通过对话逐步理清您的业务设计
2. 原型生成：基于设计生成可以直接在浏览器打开的原型

请告诉我您想做什么？
```

### 3.2 描述您的想法

您回答：

```
我想做一个餐厅点餐系统，现在顾客点菜都是服务员手写记录，
经常出错，传给厨房也容易搞混，想用系统来管理。
```

迷悟师会确认理解，并继续问问题。

### 3.3 经历两个阶段

接下来，迷悟师会引导您完成两个阶段：

**阶段一：需求建模**（通过对话逐步深入，每个话题约 2~5 分钟）

```
话题 1：了解背景  ──→  几张桌子？几个服务员？内部用还是顾客自助？
话题 2：确认角色  ──→  谁会用这个系统？（服务员、厨师、收银员……）
话题 3：梳理流程  ──→  从顾客坐下到结账离开，每步谁做什么？
话题 4：确认对象  ──→  需要管理哪些东西？（菜单、订单、桌台……）
话题 5：建立关系  ──→  订单和桌台是什么关系？一张桌可以有多个订单吗？
话题 6：梳理状态  ──→  订单有哪些状态？（已下单、制作中、已上菜……）
话题 7：确认规则  ──→  什么情况下不能修改订单？厨房确认后还能改菜吗？
```

**阶段二：原型生成**（确认页面规划后自动生成）

```
规划权限  ──→  每个角色能看到哪些页面？
规划功能  ──→  每个页面有哪些按钮和操作？
规划跳转  ──→  点击按钮后页面如何跳转？
生成代码  ──→  自动生成可在浏览器打开的原型
```

### 3.4 验证模型

需求建模完成后，迷悟师会展示一份验证报告：

```
┌─────────────────────────────────────────┐
│  模型验证报告                            │
├─────────────────────────────────────────┤
│  项目：餐厅点餐管理系统                  │
│  对象数：4  角色数：3  场景数：4         │
├─────────────────────────────────────────┤
│  结构验证：✅ 通过                       │
│  完整性：  ✅ 通过                       │
│  语义合理性：✅ 通过                     │
│  置信度：  88%（良好）                  │
└─────────────────────────────────────────┘

模型质量良好，建议进入原型生成阶段。
```

### 3.5 生成原型

确认进入原型阶段后，迷悟师会生成以下文件：

```
✅ prototype/styles.css 已生成
✅ prototype/app.js 已生成
✅ prototype/menu.html 已生成
✅ prototype/orders.html 已生成
✅ prototype/index.html 已生成

原型生成完毕！
打开方式：用浏览器直接打开 prototype/index.html
```

用浏览器打开 `prototype/index.html`，就能看到一个可以点击演示的系统原型。

## 4. 需求建模详解

这一部分帮助您理解建模过程中的每个话题在做什么，以及如何更好地回答问题。

### 步骤 0：说出您的想法

**迷悟师在问什么**：您想做什么系统？解决什么问题？有没有现成的资料？

**您需要提供**：
1. 一段简单的描述，不需要很正式，口语化就行
2. 如果有需求文档、产品说明书等资料，可以提供

**迷悟师会询问**：
> "您有没有需求文档、产品说明书或其他参考资料？"
>
> **选项**：
> - **A. 有文档**：提供 PDF、Word、Markdown 等文件的本地路径
> - **B. 有网页链接**：提供产品文档 URL
> - **C. 没有，我直接描述**：通过对话描述需求

**建议**：如果您已有需求文档或产品说明书，强烈建议提供，可以加速建模过程并提高准确性。

**好的回答示例**：
> "我们公司的差旅报销很麻烦，员工填纸质单子，财务手动核对，经常要来回确认，想做个线上的流程。"
>
> （如果选择 A）提供《差旅报销需求说明书.pdf》的路径，如 `E:\docs\差旅报销需求说明书.pdf`

**不好的回答示例**：
> "我要做一个报销系统。"（太简短，迷悟师会继续追问）

**如果您提供了资料**：
迷悟师会读取资料内容，提取关键信息（用户角色、业务对象、业务规则等），生成摘要供您确认，并在后续步骤中引用这些信息辅助建模。

**迷悟师输出**：
- 记录您的系统基本定位
- 如果有资料，还会生成资料摘要和关键信息提取记录

---

### 步骤 1：了解业务背景

**迷悟师在问什么**：这个系统给谁用？多少人用？

**典型问题**：
- "这是公司内部员工使用，还是面向外部客户的？"
- "大概多少人会用这个系统？"

**回答建议**：如实描述即可，数量不需要很精确。

---

### 步骤 2：确认使用角色和场景

**迷悟师在问什么**：谁在用这个系统？每种人分别在什么情况下用？

**示例对话**：

```
迷悟师：这个报销系统里，谁会用它？

您：员工报销，财务审核，还有部门主管审批。

迷悟师：好的，我记录了三个角色：员工、财务、主管。
对于"员工"，能描述一个最典型的工作场景吗？

您：员工出差回来，把票据照片上传，填金额和用途，
提交给主管审批，审批完财务打款。

迷悟师：明白了。这个场景开始前需要什么条件？
（比如"出差已完成"？）
```

**迷悟师输出**：角色列表和使用场景描述。

---

### 步骤 2.5：梳理跨角色流程

**迷悟师在问什么**：有没有需要多个人配合完成的完整流程？

**适用情况**：有审批链、有交接的场景（报销审批、采购流程等）。

**如果只有单人使用**（比如个人任务管理），迷悟师会自动跳过这步。

---

### 步骤 3：确认要管理的东西

**迷悟师在问什么**：系统里需要管理哪些"东西"？每个"东西"需要记录哪些信息？

这是最核心的步骤。迷悟师会从您之前的描述中，自动提取出候选清单，供您确认。

**示例对话**：

```
迷悟师：从您的描述中，我识别出这些需要管理的东西：
  • 报销单
  • 票据（照片）
  • 员工信息

对吗？有遗漏的吗？

您：差不多，但员工信息不用在这里管，
用公司已有的HR系统就行。

迷悟师：明白，那我们只管理"报销单"和"票据"。
关于"报销单"，需要记录哪些信息？

您：报销类型（差旅、招待、日常）、金额、日期、
说明、相关票据、当前状态。

迷悟师：好的，哪些信息是必须填的？
```

**迷悟师输出**：每个对象的信息字段列表。

---

### 步骤 3.5：建立关系

**迷悟师在问什么**：这些"东西"之间有什么关联？

**常见关系类型**（迷悟师会用通俗语言问，您不需要懂这些术语）：

| 关系类型 | 通俗解释 | 举例 |
|---------|---------|------|
| 一对一 | 一个 A 对应一个 B | 一个员工有一个工号 |
| 一对多 | 一个 A 对应多个 B | 一张报销单可以有多张票据 |
| 多对多 | 多个 A 对应多个 B | 一个订单包含多个商品，一个商品可出现在多个订单里 |

**示例对话**：

```
迷悟师：一张"报销单"可以包含多张"票据"，还是只能有一张？

您：可以有多张，比如出差会有火车票、住宿发票、餐饮发票。

迷悟师：如果删掉一张报销单，对应的票据怎么处理？

您：一起删掉吧，票据没有单独存在的意义。
```

---

### 步骤 4：梳理状态变化

**迷悟师在问什么**：某个"东西"在生命周期里会经历哪些阶段？怎么从一个阶段变到另一个阶段？

**示例对话**：

```
迷悟师："报销单"刚创建时是什么状态？

您：草稿状态，还没有提交。

迷悟师：它会变成哪些状态？

您：提交后变成"待审批"，主管审批后变成"待财务"，
财务处理后变成"已打款"，也可能被驳回。

迷悟师：让我确认一下状态变化：

  草稿 → 待审批（触发：提交）
  待审批 → 待财务（触发：主管审批通过）
  待审批 → 已驳回（触发：主管驳回）
  待财务 → 已打款（触发：财务完成打款）

这样对吗？
```

---

### 步骤 5：确认操作和规则

**迷悟师在问什么**：对每个"东西"能做什么操作？有什么限制条件？

**示例对话**：

```
迷悟师：对"报销单"能做哪些操作？

您：新建、提交、修改、撤回、审批通过、审批驳回、打款。

迷悟师：执行"修改"有什么限制吗？什么情况下不能改？

您：只有草稿状态可以改，提交后就不能改了。
另外，主管审批通过后也能让员工修改，
比如金额有问题退回重填。

迷悟师：明白了。做完"提交"之后，需要通知谁吗？

您：通知主管说有新报销单待审批。
```

---

### 步骤 6：模型验证

所有信息收集完成后，迷悟师会自动检查模型的完整性，输出验证报告（如 [3.4 节](#34-验证模型) 所示）。

**置信度说明**：

| 置信度 | 含义 | 迷悟师的建议 |
|--------|------|------------|
| 90%–100% | 非常好 | 直接进入原型生成 |
| 70%–89% | 良好 | 可以继续，有少量可补充项 |
| 50%–69% | 一般 | 建议先补充几个关键点 |
| 低于 50% | 需要补充 | 还有重要内容没有确认，需要继续 |

置信度不到 70% 时，迷悟师会列出具体缺少什么，引导您补充。

## 5. 原型生成

### 原型是什么

原型是一组可以直接在浏览器打开的网页，它会：

- 展示您的系统有哪些页面
- 每个页面有哪些功能按钮
- 数据以什么形式呈现（表格、列表等）
- 操作按钮点击后会发生什么变化

原型使用**模拟数据**，不连接真实数据库，主要用于：
- 向开发团队展示需求
- 向领导或投资人演示产品
- 自己验证想法是否合理

### 原型生成过程

进入原型生成阶段后，迷悟师会先和您确认三件事：

**1. 每个角色能看到哪些页面？**

```
迷悟师：基于您描述的三个角色，我规划了以下访问权限：

  员工：可以看到"我的报销单"页面
  主管：可以看到"待审批列表"和"我的报销单"
  财务：可以看到"待打款列表"

这样合理吗？
```

**2. 每个页面有哪些功能？**

```
迷悟师："我的报销单"页面的功能：
  • 查看报销单列表
  • 新建报销单
  • 查看详情
  • 撤回/修改（仅草稿状态）

这样对吗？
```

**3. 页面之间怎么跳转？**

迷悟师会展示主要的操作流程，确认无误后开始生成代码。

### 规划记录的自动保存

三项确认（权限、功能、跳转）完成后，迷悟师会把规划结果自动追加到 `conversation.md` 文件中。

这意味着：如果您后来修改了设计、需要重新生成原型，再次输入 `/mims prototype` 时，迷悟师会读取上次的规划，直接问您"是否沿用上次的规划"，无需从头再来。

### 打开原型

生成完成后，找到 `prototype/index.html` 文件，双击用浏览器打开即可。

> **注意**：某些浏览器安全设置可能阻止本地文件的 JavaScript。如果页面显示不正常，尝试用 Chrome 或 Firefox 打开。

## 6. 修改已有设计

在设计过程中随时可以说"我想修改"，迷悟师会根据修改的范围，选择合适的处理方式。

### 四种修改类型

**微小改动**（直接修改，不影响其他内容）：

- 改一个中文名称
- 修改某个字段的描述

```
您：把"报销金额"改成"报销总金额"

迷悟师：好的，已更新。
```

---

**局部修改**（影响一个对象，需要回到相关步骤确认）：

- 给某个对象增加或删除一个字段
- 修改某个操作的规则

```
您：报销单还需要增加一个"成本中心"字段，对应到具体部门的预算

迷悟师：明白，我们回到"报销单"的信息字段步骤，
把"成本中心"加进去。它是必填的吗？可以有哪些选项？
```

---

**中等修改**（需要新增对象，或者多个对象都要协调修改）：

```
您：我觉得还需要管理"预算"，每个部门有年度预算，
报销会消耗预算，超预算要额外审批

迷悟师：这是一个新增"预算"对象的变更，会影响：
  • 新增对象：预算
  • 新增关系：报销单 → 预算（关联）
  • 可能新增规则：超预算时的审批流程

我会从步骤 3（确认对象）重新开始，
已有的报销单和票据信息保留，只补充预算相关内容。
继续吗？
```

---

**重大修改**（删除已有对象，或改变整体架构）：

迷悟师会告知完整影响，并提供"在当前项目修改"或"建议新建一个项目"两个选项。

---

### 继续上次的设计

如果中途退出，下次重新输入 `/mims design`，迷悟师会检测到上次的设计文件：

```
发现上次的设计记录（报销管理系统，置信度 78%）。
是继续上次的设计，还是重新开始？
```

## 7. 常用命令速查

在 Claude Code 的对话框中输入以下命令：

| 命令 | 用途 | 什么时候用 |
|------|------|---------|
| `/mims` | 查看迷悟师介绍和可用命令 | 第一次使用，或忘记命令时 |
| `/mims design` | 启动或继续设计流程 | 开始新设计，或继续上次 |
| `/mims model` | 查看当前设计摘要 | 想看看目前整理了哪些内容 |
| `/mims validate` | 手动检查模型质量 | 想主动验证一下当前设计 |
| `/mims prototype` | 直接生成原型 | 已有设计文件，跳过建模步骤 |
| `/mims change` | 主动进入修改流程 | 知道要改什么，不想等迷悟师检测 |

## 8. 使用技巧

### 技巧一：不用一次说清楚所有内容

迷悟师会主动追问，您只需要描述自己知道的部分。如果某个问题暂时不确定，直接说"先跳过"，后面可以随时补充。

```
迷悟师：这个操作有什么约束条件？

您：这个我还没想清楚，先跳过。

迷悟师：好的，记录为待确认，我们继续下一步。
```

---

### 技巧二：用例子代替抽象描述

遇到难以描述的规则时，举一个具体例子往往比直接描述更有效：

**不清晰的描述**：
> "报销有金额限制"

**清晰的例子**：
> "500元以下主管可以直接审批，500元以上需要总监审批，2000元以上还需要财务总监审批"

---

### 技巧三：随时要求确认和调整

每个步骤结束时，迷悟师都会展示小结，请认真看一遍：

```
迷悟师：让我确认一下目前的报销单信息：

  必填：报销类型、金额、说明、提交日期
  选填：备注

这样完整吗？
```

如果有问题，及时指出，越早发现越容易修改。

---

### 技巧四：从简单开始，逐步完善

第一次使用时，建议选择一个相对简单的系统来练手（比如任务管理、联系人管理），熟悉流程后再处理复杂的业务。

---

### 技巧五：原型可以多次生成

修改设计后，可以重新输入 `/mims prototype` 生成新版本的原型。原型会覆盖 `prototype/` 目录下的旧文件。

---

### 技巧六：善用参考资料

如果您已有需求文档、产品说明书、会议记录等资料，可以在 Step 0 时提供给迷悟师。

**优势**：
- **自动提取关键信息**：迷悟师会从资料中提取用户角色、业务对象、业务规则等，减少重复描述
- **提高建模准确性**：基于文档的提取信息比口头描述更准确
- **加速对话流程**：迷悟师可以直接引用资料内容，减少确认轮次

**支持的资料类型**：
- **文档**：PDF、Word、Markdown、TXT
- **图片**：界面截图、流程图、架构图
- **网页**：产品文档 URL
- **文本**：直接粘贴的文本内容

**使用示例**：

```
迷悟师：您有没有需求文档、产品说明书或其他参考资料？

您：有，我的产品需求文档在 `E:\docs\prd.pdf`。

迷悟师：好的，我已经阅读了您的《产品需求文档》。

我从资料中提取了以下关键信息：

┌─────────────────────────────────────────┐
 │  📄 文档类型: PDF                         │
 │  🎯 业务目标: 构建订单管理系统            │
 │  👥 目标用户: 客户、管理员                │
 │  📦 核心对象: 订单、产品、客户            │
 │  📋 业务规则: 订单金额>1万需审批          │
 └─────────────────────────────────────────┘

这些信息准确吗？

您：准确，继续。

迷悟师：好的，我会在后续步骤中引用这些信息...
```

**注意事项**：
- 迷悟师记录的是资料的**本地路径**，不是复制或上传资料
- 如果资料文件被移动或删除，迷悟师将无法再次访问
- 原始资料文件不会被修改
- 提取的关键信息存储在 `domain-model.yaml` 的 `source_materials` 字段中

## 9. 常见问题

**Q：我不懂技术，能用迷悟师吗？**

可以。迷悟师专为非技术人员设计，全程使用日常语言，不会要求您理解任何技术概念。

---

**Q：输入 `/mims` 没有反应怎么办？**

检查以下几点：
1. 确认 `CLAUDE.md` 文件在您的项目根目录（不是 `.claude` 目录里）
2. 确认 `.claude/skills/mims/SKILL.md` 文件存在
3. 重启 Claude Code，重新进入项目目录后再试

---

**Q：设计做到一半，可以明天继续吗？**

可以。设计进度保存在 `domain-model.yaml` 文件里，退出 Claude Code 后不会丢失。下次输入 `/mims design` 会提示您继续上次的设计。

---

**Q：原型打开后显示空白或样式错误怎么办？**

1. 确认用的是 Chrome 或 Firefox 浏览器
2. 如果是 Safari，可能有本地文件限制，建议换 Chrome
3. 右键网页选"查看页面源代码"，如果有内容则是 CSS 问题，把 `styles.css` 和 `app.js` 是否在同一目录

---

**Q：原型里的数据是真实数据吗？**

不是。原型使用的是**模拟数据**，由迷悟师根据您的字段描述自动生成，不会连接任何真实数据库。这些数据仅用于演示。

---

**Q：可以把原型直接给开发团队用吗？**

可以把原型和 `domain-model.yaml` 一起给开发团队参考，但原型本身**不能**直接用于生产。开发团队需要根据 `domain-model.yaml` 中定义的业务规则，重新开发完整的系统。

---

**Q：生成的 `domain-model.yaml` 文件是什么，能手动修改吗？**

这个文件是迷悟师记录所有业务设计的地方，格式规整、人类可读。您可以用文本编辑器打开查看，也可以手动修改，修改后用 `/mims validate` 命令检查有没有格式错误。

---

**Q：我的需求很复杂，有几十个功能，迷悟师能处理吗？**

建议把复杂系统拆分成几个子系统，每次设计一个模块。比如"电商系统"可以拆成"商品管理"、"订单管理"、"用户管理"分别设计，最后由开发团队整合。

---

**Q：设计完成后，怎么把结果给开发团队？**

将以下文件打包发给开发团队：
- `domain-model.yaml`：业务模型定义（开发团队可直接用）
- `prototype/` 目录：可交互原型（用于演示和沟通）
- `conversation.md`：完整对话记录（包含所有设计决策的背景）

---

**Q：提供的资料会被如何使用？**

迷悟师会：
1. **读取资料内容**：使用 AI 能力理解文档、图片或网页内容
2. **提取关键信息**：识别用户角色、业务对象、业务规则等13类信息
3. **生成结构化摘要**：汇总关键信息供您确认
4. **辅助后续建模**：在 Step 1-6 中引用资料内容，提高准确性和效率

所有提取的信息都会标注来源（如"根据您的《产品需求文档》第2页..."），您可以在 `domain-model.yaml` 的 `source_materials` 字段中查看完整的提取记录。

---

**Q：资料中的敏感信息如何处理？**

迷悟师会：
- **仅提取业务相关信息**：只提取与系统设计相关的业务信息（角色、对象、规则等）
- **不记录敏感数据**：不会将具体的人名、电话、账号等敏感数据写入 `domain-model.yaml`
- **本地路径记录**：只记录资料的本地路径，不复制或上传资料文件

---

**Q：可以提供多个资料吗？**

可以。您可以：
- 提供多个文档路径（如 PRD、会议记录、产品说明书）
- 同时提供文档和网页链接
- 在不同阶段补充新的资料

迷悟师会合并提取所有资料的信息，并在提取记录中标注每条信息的来源。

---

**Q：如果资料内容与我的描述冲突怎么办？**

迷悟师会：
1. **优先询问您**：展示资料提取的内容，询问是否准确
2. **以您的确认为准**：您可以在确认时修改或补充
3. **记录冲突**：如果资料内容与您的口头描述不一致，会记录下来供后续参考

## 附录：需求说明抽屉使用说明

### 功能介绍

每个原型页面底部都有一个**需求说明抽屉**（Requirements Drawer），用于展示与当前页面相关的功能需求信息，方便对照讲解和评审说明。

### 如何使用

1. **展开抽屉**：点击页面底部的"📋 需求说明"栏
2. **查看需求**：在5个标签页中切换查看不同类型的需求
3. **收起抽屉**：点击"收起"按钮或再次点击"📋 需求说明"栏

### 标签页说明

| 标签页 | 内容 | 用途 |
|--------|------|------|
| **场景** | 使用场景描述 | 了解用户在什么情况下使用此功能 |
| **对象** | 业务对象定义 | 查看对象的属性和状态 |
| **操作** | 可用操作列表 | 了解可以执行哪些操作 |
| **规则** | 业务规则说明 | 查看操作的限制条件 |
| **流程** | 业务流程步骤 | 了解跨角色的协作流程 |

### 使用场景

**场景1：需求讲解**
- 展开抽屉，切换到"场景"标签
- 对照场景描述讲解原型功能
- 切换到"操作"标签，说明可执行的操作

**场景2：开发评审**
- 切换到"对象"标签，查看对象属性
- 切换到"规则"标签，确认业务规则实现
- 切换到"流程"标签，验证流程完整性

**场景3：需求变更**
- 展开抽屉查看原始需求
- 对照变更内容确认影响范围
- 记录需要修改的部分

### 注意事项

- 抽屉默认折叠，不遮挡页面主内容
- 展开时最大高度为屏幕的40%，避免影响浏览
- 需求数据来自 `domain-model.yaml`，确保一致性
- 如需查看完整需求，建议同时打开 `domain-model.yaml` 文件
