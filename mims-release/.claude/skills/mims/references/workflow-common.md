# workflow-common.md — 设计过程共性交互机制

> 本文档定义跨所有设计阶段（初步设计、详细设计、原型）的共性交互机制。
> 在设计流程启动时确定性加载，与 `schema.md` 同级。

---

## 一、信息补充提问

每步用户确认后，检测信息缺口并追问。

### 触发信号

| 信号 | 检测方式 | 处理 |
|------|---------|------|
| 多次"不确定""大概""好像" | 当前步骤内累计 ≥2 次 | "没关系，先记录初步想法。您能多描述一下具体情况吗？" |
| 本步信息量明显少于前一步 | 对比前后步骤用户描述的详略程度 | "关于这一点，您能再多说一些吗？比如具体的步骤或判断标准？" |
| 用户说"还有其他想法" | 直接表达 | "请说，我记下来。" |
| 阶段切换时（P6→D1、D5→R1） | 结构性检查点 | "在进入下一阶段之前，有没有遗漏或需要调整的地方？" |

### 原则

- 开放式引导，不给压力
- 不因信息不足阻止流程推进，先记录再补充
- 阶段切换时做一次全面回顾

## 二、思路整理引导

与"信息补充"不同——用户**不是没有信息**，而是信息太散、决策未定。追问反而加重负担。正确做法是**建议用户暂时离开设计流程**，独立把想法写清楚，回来后再提交文档让 Agent 读取。

### 触发条件（模式信号，单次不触发）

| 信号 | 检测方式 |
|------|---------|
| 反复中断恢复 | 同一阶段 `design_progress` 显示多次 resume，或同一 unfinished step 被重复进入 ≥2 次 |
| 对同一问题反复讨论 | 同一步骤内，用户对同一要素提出 ≥3 次修改或不同说法 |

### 触发后的引导话术

```
我注意到关于【{当前步骤主题}】，咱们的讨论来回了几次。
这往往说明想法还没完全定型，与其在对话中反复调整，不如
您先独立整理一下思路——可以写在文档里，不限格式，
把您确定的想法、还在犹豫的点都写下来。

等您理清楚了，随时回来输入 /mims design 继续就行。
您可以把写好的文档路径告诉我，我来读取和分析。
```

### 用户回来后的处理

1. 用户执行 `/mims design` → 恢复到上次步骤
2. 用户提供文档路径 → 以插入式方式执行 P2（资料理解）流程读取文档，提取的信息融入当前步骤，不影响 `design_progress` 进度标记
3. Agent 提取文档中的关键信息，融入当前步骤继续推进
4. 不重复引导——同一问题触发过一次后不再二次提示

### 与 P2 的衔接

P2（资料理解）本身就支持读取用户文档。思路整理引导产出的文档，在用户回来后作为 P2 的输入自然接入，不需要额外机制。若文档已经具备结构化 SRS 特征，应执行下方“采纳-精炼模式”，先让用户确认是否采纳，再决定是否写入 `requirements.srs`。

### 原则

- 语气是关心而非质疑——"与其反复调整，不如先理清楚"
- 用户选择继续对话则尊重，不强制暂停
- 不评判用户整理的文档质量，只提取有价值的信息

## 三、采纳-精炼模式

当用户提供结构化需求文档、长篇需求材料，或明确表示“按这个文档来”“在这个基础上继续完善”时，Agent 应区分四类内容：

| 类型 | 写入位置 | 说明 |
|------|----------|------|
| 来源证据 | `source_materials` | 原文摘要、提取项、来源页码/段落，忠实保存 |
| SRS 候选 | `requirements.srs.candidates[]` | 可进入需求说明书的候选需求，需用户确认采纳状态 |
| 模型事实 | `function` / `domain` / `external_interfaces` / `non_functional` | 已确认的结构化设计事实，仍是生成文档的依据 |
| 待确认项 | candidate `status: needs_clarification` 或 validation warning | 不作为已确认需求写入正式 SRS 正文 |

### 触发信号

- 文档包含功能需求章节、FR-xxx 编号或等价结构
- 文档描述了用户角色、场景、流程
- 文档覆盖接口、数据、非功能需求、约束或假设
- 用户要求“不要重新生成一份，直接在这个文档上完善”

### 对话原则

1. **先评估再询问**：说明文档覆盖了哪些 SRS 要素，再询问是否采纳为 `srs.md` 初稿来源。
2. **不覆盖原文档**：原始资料保留为来源证据，正式产物始终保存为 `{location}/srs.md`。
3. **不绕过验证**：采纳模式下，P6 仍必须通过 preliminary validation 后才能生成或精炼正式 SRS。
4. **不混淆 SRS/SDD**：只有影响需求范围、角色、场景、流程、接口、权限、验收或非功能承诺的内容回写 SRS；对象属性、状态、操作、规则等详细设计优先写入 SDD。
5. **阶段完成后补充资料**：如果 preliminary 已完成，新的资料采纳不得直接覆盖已完成阶段；应走 `/mims change` 或委托 `mims-change-manager` 分析影响层后再同步 SRS/SDD。

### 推荐话术

```
您的文档已经覆盖了需求说明书的主要结构：{覆盖摘要}。
我可以在后续设计中直接在此基础上更新和精炼它，把它打磨成正式的 srs.md；
原始文档会保留不覆盖，正式产物保存到 {location}/srs.md。

您希望采用这种方式吗？
A. 采纳这份文档作为 SRS 初稿来源
B. 只把它作为参考资料，仍由 MIMS 在 P6 生成 SRS
C. 先逐条确认其中哪些需求要采纳
```

## 四、全局进度感知

**每次阶段切换和检查点时，展示全局进度条**：

```
设计进度：[████████░░░░░░░░░░░░] 40%

✅ 初步设计（已完成）
   P1 收集 → P2 资料 → P3 背景 → P4 角色/场景 → P5 流程 → P6 架构
✅ 详细设计（已完成）
   D1 对象 → D2 关系 → D2.5 验证 → D3 状态 → D4 操作 → D4.5 覆盖 → D5 验证
→  原型生成（进行中）
   R1 分析 → R2 权限 → R3 功能 → R4 流程 → R5 结构 → R6 交互 → R7 生成 → R8 验证 → R9 交付
```

**展示时机**：
- P6 检查点通过后（进入详细设计前）
- D5 检查点通过后（进入原型前）
- R9 交付完成后

**原则**：让用户始终知道"走了多远、还有多少、接下来做什么"。

## 五、质量快照与置信度决策

### 质量快照

每个步骤完成后，展示当前步骤的质量状态：

```
完成度：[████████░░] 80%
• ✅ 已确认的对象：3个
• ✅ 已定义的关系：2个
• ⚠️ 待补充：订单的状态流转尚未定义
• 💡 【内容推荐】的对象属性以非结构化文本为主，建议后续交由专门的内容管理工具处理
```

**格式规则**：
- 用进度条直观展示完成度
- ✅ 列出已完成项（数量或具体名称）
- ⚠️ 列出待补充项（具体描述缺什么）
- 💡 列出适用性边界提示（检测到阻力信号时）
- 不超过 6 行，保持简洁

**各步骤检查内容**（详见表）：

| 步骤 | 检查项 | 不通过时 |
|------|--------|---------|
| P4 后 | 每个角色至少有 1 个场景；场景有前/后置条件 | 列出缺失场景的角色，引导补充 |
| P5 后 | 跨角色流程的步骤和角色已明确；或确认无跨角色流程 | 列出未明确的流程步骤 |
| D1 后 | 每个对象属性 ≥2；每个属性有类型；有 primary_key；附件需求已判断为 string 字段或独立对象 | 列出缺失属性或附件归属不清的对象，引导补充 |
| D2 后 | 无孤立对象；关系基数已明确；所有对象归属模块 | 列出未关联对象，引导建立关系 |
| D2.5 后 | 跨阶段交叉验证无 ERROR | 引导修复 ERROR 后再进入 D3 |
| D3 后 | 有生命周期的对象有且仅有 1 个初始状态；终止状态无出边；所有转移的操作名已在对象中定义 | 列出问题，回到 D3 修复 |
| D4 后 | 所有对象已识别操作；关键操作有至少 1 条规则；有 success 响应；附件上传/预览/下载/删除/替换及限制规则已补齐；permissions 已回填 | 列出缺规则、未识别操作或附件操作不完整的对象，引导补充 |
| D4.5 后 | P 阶段语义承诺覆盖率检查完毕；高置信度疑似遗漏已处理或确认接受；附件语义已落到字段/对象/操作/规则 | 列出未覆盖的语义要素，引导回填到属性/规则/操作/响应 |
| D5 | 委托 `mims-validator` 执行完整验证 | 按置信度四级决策 |

### 适用性阻力信号检测

在建模过程中（P3 → D5），持续检测用户需求是否超出 MIMS 的建模能力范围。检测到信号时以 `💡 提示` 级别附在质量快照中。

#### 5 个阻力信号

| # | 信号 | 检测时机 | 检测方式 | 反馈话术 |
|---|------|---------|---------|---------|
| 1 | **对象属性模糊** | D1 | 对象的属性反复修改（≥ 3 次），或属性多为 text 类型且无明确业务含义 | "关于{对象}需要记录的信息，似乎边界还不太清晰。这在{可视化/内容创作}类系统中比较常见——核心需求可能更多在于**呈现方式**而非**数据结构**。我先帮您把确定的部分记录下来，不确定的标注为待定。" |
| 2 | **对象爆炸** | D1/D2 | 单模块对象 > 8 且关联松散（超过半数对象无直接关系） | "我发现{模块}里要管理的东西比较多，彼此关联不太紧密。这可能意味着系统覆盖范围偏大。要不要先聚焦最核心的几个？" |
| 3 | **流程碎片化** | P5 | 步骤间缺少明确触发关系，或角色模糊（≥ 2 个步骤无法确定 actor） | "这个流程的步骤之间衔接关系不太明确。这通常意味着流程本身还在探索阶段。我们可以先用简化版本记录整体意图，具体步骤等想法更清晰后再补充。" |
| 4 | **规则嵌套过深** | D4 | 单条规则条件嵌套 ≥ 3 层，或规则涉及概率/阈值优化而非确定性条件 | "这条规则涉及的条件比较复杂，可能已经属于算法层面的问题了。我们可以用简化的业务语言先记录意图（比如"根据用户行为自动推荐"），具体实现留给开发团队来设计。" |
| 5 | **模块边界反复调整** | P6/D2 | 模块职责在讨论中被拆合 ≥ 3 次，或单场景跨 3+ 模块 | 复用 P6 功能-结构背离检测话术 |

#### 信号升级规则

| 情况 | 处理方式 |
|------|---------|
| 首次出现 | `💡 提示` 级别，附在质量快照中 |
| 同一维度信号重复出现 ≥ 2 次 | 独立对话轮次，展开讨论是否调整范围或拆解需求 |
| ≥ 2 个不同维度信号同时出现 | 主动建议暂停当前步骤，重新审视整体适用性 |

#### 升级后的话术（第二档）

```
我注意到在{列举已出现的信号}方面遇到了一些困难。
这可能说明系统的某些部分超出了我最擅长的范围。

我建议我们：
1. 先聚焦在{明确可以建模的部分}继续推进
2. {超出范围的部分}标注为"待后续处理"，记录需求意图但不深入建模
3. 最终转交开发团队时，这部分会作为独立的需求说明

您觉得这样安排合适吗？
```

---

### 四级置信度决策

| 置信度 | 行为 |
|--------|------|
| ≥90% | 直接建议进入下一步 |
| 70–90% | 展示可补充项，询问"继续还是补充？" |
| 50–70% | 列出建议补充的问题，引导选择 |
| <50% | 必须补充后才能继续，列出必须解决的缺口 |

**验证未通过时的引导话术**：
- 置信度 50-70%："我检查了一下，发现还需要补充以下内容：{逐条列出}。我们先从第一个开始？"
- 置信度 <50%："要继续往下走，有几个关键问题需要先解决：{逐条列出}。"
- 每次补充后重新验证，不跳过未解决的问题。

## 六、阶段过渡引导

每次从一个阶段进入下一阶段时，按以下模板引导用户：

### 模板结构

1. **肯定成果**：简要总结已完成的内容和产出
2. **预告下一步**：用非技术语言描述即将做什么、大概需要多久
3. **最后确认**：询问是否有补充或调整

### 话术模板

```
{当前阶段}完成了！{一句话总结成果，如"我们已经理清了系统的角色、场景和流程"}。

接下来我们进入{下一阶段名称}——{用日常语言描述做什么，如"逐个明确系统要管理的内容、它们之间的关系和运作方式"}。
{补充说明，如"这个阶段大概需要讨论几个对象，我会一个一个带您梳理"}。

如果有需要补充或调整的，现在是最好的时机。准备好了吗？
```

### 各过渡点的具体话术

**P6 → D1（初步设计 → 详细设计）**：
> "初步设计完成了！接下来我们会进入详细设计阶段，逐个明确系统要管理的内容、它们之间的关系、状态变化和操作规则。这个阶段大概需要讨论几个对象，我会一个一个带您梳理。如果有需要补充或调整的，现在是最好的时机。准备好了吗？"

**D5 → R1（详细设计 → 原型生成）**：
> "详细设计也完成了！接下来进入最后一步——原型生成。我会根据前面的设计，生成可以在浏览器里直接打开的原型页面，您可以实际操作体验。这一步主要由我来完成，需要您确认的时候我会提问。我们开始吗？"

**R9 完成后**：
> "原型已全部生成！您可以在浏览器中打开 `prototype/index.html` 体验。发现需要调整的地方随时告诉我。"

### 原则

- 每次过渡只做一次，不重复引导
- 语气是邀请而非命令
- 给用户选择权——可以补充、调整，也可以直接继续

## 七、转交开发指引

当用户提到"转交开发""交给开发团队""开发接手""怎么给开发"等意图时，提供以下指引。

### 触发条件

用户明确表达要将设计成果交给开发团队实施。

### 指引话术

```
好的，设计成果已就绪。以下是交付给开发团队的文件和阅读建议：

交付文件：

| 文件 | 内容 | 开发者看什么 |
|------|------|------------|
| domain-model.yaml | 完整领域模型（结构化数据） | 数据库设计、字段类型、校验规则、状态机、业务规则 |
| srs.md | 软件需求规格说明书 | 项目背景、用户角色、业务场景、业务流程、外部接口 |
| sdd.md | 软件设计规格说明书 | 业务对象关系、状态生命周期、操作与规则、模块归属 |
| prototype/ | 可交互 HTML 原型 | 界面布局、交互方式、页面流程、导航结构 |

建议开发团队按以下顺序阅读：
1. srs.md — 理解要做什么（业务背景和需求）
2. prototype/ — 体验最终效果（打开 index.html）
3. sdd.md — 理解怎么设计（对象、状态、规则）
4. domain-model.yaml — 精确定义（字段类型、校验、关系）

如果开发团队有疑问，可以随时让我帮忙解释设计中的任何部分。
```

### 原则

- 不生成额外文件，现有 4 份产出物已足够开发使用
- 指引的重点是告诉开发者**从哪读什么**，而非重复已有内容
- 可根据用户的具体开发团队情况（技术栈、团队规模等）调整阅读建议
