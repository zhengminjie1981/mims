# 设计变更参考数据

> **用途**：本文件为 mims-change-manager 子代理提供参考数据，由 SKILL.md 加载并传入子代理。
> **定位**：变更级别定义、映射表、判定规则、记录模板。不含执行流程（执行流程见 mims-change-manager.md）。
> **适用范围**：设计过程中（P/D/R 阶段）及原型后的统一变更管理。

---

## 触发条件

以下方式均可触发变更：

**设计过程中**：用户说"修改"/"改一下"/"不对"/"还有"等词语时。

**原型后反馈**：用户基于原型体验提出修改意见时（体验引导见 `workflow-prototype.md` R9）。用户通过口头描述反馈，包含页面名、操作和问题描述。

**Agent 检测触发**：建模过程中发现以下问题：
- 发现新对象与已有对象存在未声明的关系
- 检测到规则冲突（两条操作规则互相矛盾）
- 发现引用断裂（某操作/场景引用了不存在的对象或状态）
- 用户描述与已建模内容矛盾

Agent 检测触发时，先向用户说明问题，再询问是否进入变更流程。

## 反馈分类（原型后触发时）

原型后反馈需先分类，设计过程中的变更跳过此步直接进入变更级别识别。

| 类型 | 特征 | 例子 |
|------|------|------|
| **简单反馈** | 不涉及业务逻辑变更，只影响 UI 表现 | 改标签文案、调整布局位置、改颜色/图标、字段显示顺序、按钮位置 |
| **复杂反馈** | 涉及业务逻辑、数据结构、流程或角色的变更 | 审批流程不对、缺少功能页面、角色权限不合理、业务规则需调整 |

**判断原则**：反馈内容如果需要修改 `domain-model.yaml` 的 F 层（场景/流程）或 S/B 层（对象/状态/操作/规则），则为复杂反馈。

**简单反馈处理**：主代理直接修改原型文件 + 检查是否需同步模型（如标签文案对应 `chinese_name`）。不走变更管理流程。

## 变更级别（统一）

变更级别按**影响的数据层**判断，而非当前所在阶段。

| 级别 | 影响范围 | 判断标准 | 回退策略 |
|------|---------|---------|---------|
| **L1 轻微** | 仅改名/描述 | 改名称、描述、中文名、标签、文案 | 当前步骤直接修改，无需回退 |
| **L2 局部** | 单一元素 | 改单个元素：角色属性、场景条件、流程步骤、单对象的属性/关系/操作/规则 | 回退到该元素的建模步骤 |
| **L3 中等** | 多元素协调 | 新增元素（角色/场景/流程/对象），或多元素协调修改 | 回退到**受影响最早的阶段**起点，保留已有内容 |
| **L4 重大** | 架构/核心 | 删除核心元素，或改变整体架构/核心场景/核心角色 | 展示完整影响分析，提供"新建项目"或"当前重建"选项 |

**受影响最早的阶段**判定：

| 变更影响 | 受影响的最早阶段 | 回退起点 |
|---------|----------------|---------|
| 仅 B 层（状态/操作/规则） | 详细设计 | D1 |
| 仅 S 层（对象/属性/关系） | 详细设计 | D1 |
| S + B 层 | 详细设计 | D1 |
| F 层（场景/流程）+ S/B 层 | 初步设计 | P3 |
| 仅 F 层 | 初步设计 | P3（或该元素的建模步骤） |

## 按阶段预检

### P 阶段预检（仅有 F 层数据）

| 检查项 | 规则 |
|--------|------|
| 角色被引用 | 变更后每个 actor 仍在至少 1 个 scenario 的 actors[] 中 |
| 场景完整 | 变更后每个场景仍有前/后置条件 |
| 流程步骤完整 | 变更后每个流程仍有 ≥2 步 |
| 流程归属场景 | 变更后每个 process 的 parent_scenario 仍有效 |
| 模块非空 | 变更后每个模块仍有职责定义 |

### D 阶段预检（叠加 P 阶段检查）

| 检查项 | 规则 |
|--------|------|
| 孤立对象 | 对象删除后，与该对象有关的 relationship 将断裂 |
| 断裂引用 | operations/rules 引用了被删除对象或状态 |
| 状态闭环 | 变更后每个有状态的对象仍有完整的状态机（初始态、终止态） |
| 规则覆盖 | 变更后关键操作仍有至少 1 条规则 |
| 模块归属 | 变更后每个对象仍归属于某个模块 |

### R 阶段预检（叠加 D 阶段检查）

| 检查项 | 规则 |
|--------|------|
| 页面覆盖 | 变更后每个模块仍有对应页面 |
| 流程连通 | 变更后页面跳转关系仍支持流程步骤 |
| 按钮可见性 | 变更后操作按钮仍符合状态转移规则 |

## 文档同步规则

按**变更影响的数据层**判断需要同步的文档，而非当前所在阶段。

| 影响层 | 需同步的文档 |
|--------|------------|
| 仅 F 层 | srs.md |
| 仅 S/B 层 | sdd.md |
| F + S/B 层 | srs.md + sdd.md |
| 涉及原型（原型已存在） | + 原型调整（见下方） |

## 变更后流程恢复

| 级别 | 恢复策略 |
|------|---------|
| L1 | 回到当前步骤继续，无需重新验证 |
| L2 | 从回退步骤重新执行（已有内容保留），完成后重新验证 |
| L3 | 从受影响最早阶段起点重新执行，已有内容保留供参考；完成后执行跨阶段验证 + 对应阶段检查点验证 |
| L4 | 用户选择：当前重建（从 P3 重新走）或 新建项目 |

**对象删除级联清理清单**：

删除一个对象时，必须同步清理以下引用：

| 引用位置 | 清理操作 |
|---------|---------|
| `domain.objects[].relationships[]` | 移除 to 指向被删对象的 relationship；移除被删对象自身的 relationships |
| `domain.objects[].operations[].rules[].ref` | 移除引用被删对象 rules 的 ref |
| `domain.objects[].transitions[]` | 移除被删对象的 transitions |
| `domain.modules[].relationships[]` | 移除 from/to 包含被删对象的 N:M 关系 |
| `function.scenarios[].workflow[]` | 标注涉及被删对象的步骤为"待调整"，请用户确认 |
| `function.processes[].steps[]` | 同上 |
| `external_interfaces[].related_operations[]` | 移除引用被删对象操作的条目 |

## 原型自动调整

> **核心规则**：设计变更完成后，若原型已存在（R7+ 已完成），自动触发原型调整。无需用户手动请求。

### 触发条件

设计变更执行完成 + 重新验证通过 + 原型状态为"已生成"（`prototype/` 目录存在）。

### 原型影响映射

对照变更记录中的实际变更内容，判断受影响的页面：

| 变更类型 | 对原型的影响 |
|---------|------------|
| 新增/删除属性 | 列表列、表单字段、详情项 |
| 新增/删除状态 | 状态标签、操作按钮、状态机数据 |
| 新增/删除操作 | 页面按钮、确认弹窗 |
| 新增/删除对象 | 新增/移除管理页面 |
| 流程变更 | 工作台流程图 |
| 角色权限变更 | 侧边栏可见性、页面访问权限 |
| 业务规则变更 | 表单校验、操作确认逻辑 |
| 场景/角色新增 | 新增页面 + 导航 + 权限 |

### 调整规模判定

| 调整规模 | 判断标准 | 处理方式 |
|---------|---------|---------|
| **增量调整** | 变更影响 ≤3 个页面，且页面结构不变 | 主代理直接修改受影响的 HTML 文件 |
| **部分重新生成** | 变更影响 >3 个页面，或页面结构有变化 | 提示用户确认后，重新生成受影响的页面 |
| **全量重新生成** | 核心场景/角色变化，或大部分页面受影响 | 提示用户确认后，委托 mims-prototyper 全量重新生成 |

### 原型规划重新审视（R1-R6 已完成但未生成时）

| 变更涉及 | 需重新审视的规划步骤 |
|---------|-------------------|
| 角色/权限变更 | R2（页面权限规划） |
| 场景/操作变更 | R3（页面功能映射） |
| 流程变更 | R4（页面流程设计） |
| 对象/属性结构变更 | R5（页面结构设计） |
| 状态/操作/规则变更 | R6（页面交互设计） |

标记方式：将受影响的步骤从 `design_progress.prototype.completed_steps` 中移除。

## 变更记录格式

> 变更记录追加到 `sdd.md` 关键决策记录章节。同时是原型调整的输入依据。

**设计过程中触发**：

```markdown
## 变更记录 #{n} — {datetime}

- 变更级别：L{n}（{轻微/局部/中等/重大}）
- 触发方式：{用户主动 / Agent检测}
- 影响层：{F层 / S层 / B层 / F+S层 / F+S+B层}
- 变更内容：
  - {元素A：{具体修改了什么}}
  - {元素B：{具体修改了什么}}
- 影响范围：{受影响的角色/场景/对象/关系/操作列表}
- 文档同步：{srs.md / sdd.md / srs.md+sdd.md}
- 原型状态：{未开始 / 规划已完成 / 已生成}
- 原型处理：{无需处理 / 标记重新审视 R{n}/R{n} / 增量调整{页面列表} / 部分重新生成{页面列表} / 全量重新生成}
- 用户确认：{是 / 否（L1/L2 不需要确认）}
```

**原型后触发**：

```markdown
## 原型迭代 #{n} — {datetime}

- 反馈来源：{基于原型操作体验 / 基于原型截图标注}
- 反馈类型：{简单反馈 / 复杂反馈}
- 反馈内容：{逐条列出}
- 变更级别：L{n}（复杂反馈时）
- 执行结果：
  - {设计变更：{元素A 修改了什么}}
- 文档同步：{srs.md / sdd.md / 无}
- 原型处理：{简单直接修改 / 增量调整{页面列表} / 部分重新生成{页面列表} / 全量重新生成}
- 用户确认：是
```

### 重复变更识别：评估结论

**当前决定：不实现**。

理由：
1. MIMS 目标用户是非技术人员，他们提出重复变更（如再次提出之前拒绝的修改）是正常的思考过程，不应被标记为"重复"
2. 记录"之前拒绝的原因"需要在对话上下文中维护变更决策历史，增加 Agent 记忆负担
3. 简单场景下收益低——用户通常不会频繁提出重复变更
4. 如果未来发现用户频繁提出相同变更导致效率问题，可在 change-manager 中增加"变更历史回顾"步骤

### 批量变更合并分析：评估结论

**当前决定：不实现**。

理由：
1. 非技术用户通常一次提一个修改，很少同时提出多个不相关的变更
2. 如果用户连续提出多个变更，主代理可以逐个委托 change-manager 分析，最后合并影响范围
3. 合并分析需要复杂的冲突检测（变更A新增字段与变更B删除同一字段），实现成本高
4. 原型评审后如果有多条反馈，可按优先级逐条处理，不需要并行分析

### 原型调整失败处理

当设计变更完成但原型自动调整失败时（如 HTML 文件被手动修改、文件损坏），按以下流程处理：

1. **检测失败**：调整后重新执行 R8 验证，如果页面无法正常渲染或关键功能缺失，判定为调整失败
2. **提示用户**："原型自动调整遇到了问题，可能是因为文件被手动修改过。建议重新生成受影响的页面。"
3. **提供选项**：
   - A. 重新生成受影响的页面（推荐） — 委托 mims-prototyper 重新生成
   - B. 保持当前原型，仅记录设计变更 — 不修改原型文件
4. **用户选择 A**：重新生成后执行 R8 验证
5. **用户选择 B**：在 metadata 中标记"原型与设计不同步"，下次 /mims prototype 时提醒用户
