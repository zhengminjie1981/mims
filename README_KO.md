# MIMS - Make Idea Make Sense

> **아이디어를 합리적이고, 명확하며, 실행 가능하게 만듭니다**

---

## 프로젝트 개요

MIMS는 "迷悟师"(MIMS 가이드)의 형태를 취하는 AI 에이전트로, 비기술 사용자가 주도적인 다자간 대화를 통해 소프트웨어 설계를 안내하고, 막연한 아이디어를 명확한 도메인 모델과 인터랙티브 프로토타입으로 변환하도록 돕습니다.

MIMS는 Claude Code CLI 및 Codex App 사용자를 위해 배포되며, 세 가지 핵심 구성 요소로 이루어집니다: 메인 에이전트에 주입되는 페르소나 구성(`CLAUDE.md` / `AGENTS.md`), 구조화된 워크플로우(Skill), 그리고 원자적 작업을 처리하는 하위 에이전트.

### 핵심 개념

**Make Idea Make Sense** - 구조화된 대화와 AI 지원을 통해 아이디어를 프로토타입으로 변환:

- **아이디어를 명확하게** - 막연한 요구사항을 체계적이고 이해하기 쉬운 개념으로 변환
- **아이디어를 가시화** - 추상적인 개념을 시각화하고 구체화
- **아이디어를 실현 가능하게** - 아이디어를 인터랙티브 프로토타입으로 변환

### 대상 사용자

- **프로덕트 매니저** - 제품 아이디어를 빠르게 검증
- **비즈니스 분석가** - 비즈니스 요구사항을 명확하게 표현
- **창업자** - MVP(최소 기능 제품) 기획
- **도메인 전문가** - 기술 배경지식 없이 소프트웨어 설계

### 주요 특징

1. **제로 기술 장벽** - 일상 언어로 대화, 배워야 할 도구 없음
2. **AI 주도 안내** - 에이전트가 주도적으로 질문하여 요구사항 명확화
3. **자동 모델 생성** - 대화에서 도메인 객체 모델 자동 구축
4. **즉시 시각화** - 모델 변경과 관계를 실시간으로 표시
5. **인터랙티브 프로토타입** - 브라우저에서 바로 실행되는 프로토타입 생성

## 프로젝트 이름 의미

**MIMS** = **M**ake **I**dea **M**ake **S**ense

**발음**: /mɪmz/

**전체 이름**: Make Idea Make Sense

- **Make** - 만들다, 구축하다, ...게 만들다
- **Idea** - 생각, 개념, 요구사항, 영감
- **Make Sense** - 합리적이 되다, 명확해지다, 체계적이 되다, 이해 가능해지다

## 작동 방식

```
┌─────────────────────────────────────────────────────────────┐
│                     MIMS 대화 루프                          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  사용자가 아이디어/요구사항 제시                              │
│         ↓                                                   │
│  AI 에이전트가 질문 (명확화, 정제, 보완)                     │
│         ↓                                                   │
│  사용자 응답                                                │
│         ↓                                                   │
│  에이전트가 도메인 모델 자동 업데이트                        │
│         ↓                                                   │
│  모델을 사용자 확인을 위해 표시                              │
│         ↓                                                   │
│  사용자 피드백/확인                                          │
│         ↓                                                   │
│  모델이 명확하고 완전해질 때까지 루프 계속                   │
│         ↓                                                   │
│  인터랙티브 프로토타입 생성                                  │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**모델링 프레임워크**: MIMS는 설계 과학의 고전적 온톨로지인 **FBS(Function-Behavior-Structure)** 프레임워크를 채택합니다(Gero, 1990). 이는 세 가지 핵심 UML 다이어그램 유형(유스케이스/상태/클래스 다이어그램)과도 높은 대응성을 가집니다.

| FBS 계층 | 핵심 질문 | 대화 단계 |
|-----------|---------------|----------------|
| **F 기능** | 누가 사용하는가? 어떤 시나리오인가? 어떤 작업인가? | 역할 및 시나리오 분석 |
| **B 동작** | 어떤 상태인가? 어떻게 조작하는가? 어떤 규칙인가? | 상태 및 동작 모델링 |
| **S 구조** | 무엇을 관리하는가? 어떤 정보인가? 어떤 관계인가? | 비즈니스 객체 및 관계 모델링 |

> 대화 순서: 하향식(F→B→S), 모델 출력: 의존성 순서(S→B→F).

## 기술 아키텍처

```
대화 계층          데이터 계층       생성 계층           문서 계층
CLAUDE.md          YAML 모델        HTML/JS 프로토타입   Markdown 문서
+ Skill
+ 하위 에이전트
```

### 기술 스택

| 계층 | 기술 | 선택 이유 |
|-------|-----------|-----------|
| 페르소나 및 기본 동작 | CLAUDE.md / AGENTS.md | 메인 에이전트에 T0 주입, 항상 활성 |
| 대화 워크플로우 | AI CLI Skill | 온디맨드 로딩, 표준화된 트리거 |
| 프로토타입/검증 작업 | AI CLI 하위 에이전트 | 원자적 작업 격리, 컨텍스트 오염 없음 |
| 데이터 저장 | YAML | 사람이 읽기 쉽고, AI 친화적 |
| 프로토타입 생성 | HTML/CSS/JS | 종속성 없음, 브라우저에서 바로 실행 |
| 문서 출력 | Markdown | 읽기 쉽고, 버전 관리 용이 |

## 세 가지 단계

### Phase 1: 예비 설계

대화를 통해 완료 (6단계):
1. **P1 요구사항 수집 및 준비** - 초기 아이디어 이해, 자료 준비 안내
2. **P2 자료 이해** (선택) - 사용자 제공 자료 분석
3. **P3 컨텍스트 및 목표** - 비즈니스 배경 및 전반적 목표 이해
4. **P4 역할 및 시나리오** - 사용자 역할 및 유스케이스 정의 (다대다)
5. **P5 비즈니스 프로세스** - 핵심 프로세스 매핑 (시나리오 기반, 정보 흐름)
6. **P6 아키텍처 개요** - 모듈 분할, 외부 인터페이스, AI 에이전트 평가

**체크포인트**: 예비 검증 → `srs.md` 생성

**산출물**: `domain-model.yaml` (F 계층 + 모듈) + `srs.md` (소프트웨어 요구사항 명세서)

### Phase 2: 상세 설계

심층 모델링 (5단계):
1. **D1 비즈니스 객체 인식** - 관리할 대상 및 속성 추출
2. **D2 관계 및 모듈 할당** - 관계 정의, 모듈 할당, AI 에이전트 설계
3. **D3 상태 및 수명 주기** - 객체 상태 및 전이 조건 정의
4. **D4 동작 및 비즈니스 규칙** - 동작 및 비즈니스 규칙 정의
5. **D5 모델 검증 및 신뢰도** - 모델 완전성 검증, 신뢰도 평가

**체크포인트**: 전체 검증 → `sdd.md` 생성

**산출물**: `domain-model.yaml` (완전한 FBS) + `sdd.md` (소프트웨어 설계 문서)

### Phase 3: 프로토타입 생성

모델에서 생성 (9단계):
1. **R1 모델 분석** - 모델 데이터 특성 분석
2. **R2 페이지 권한** - 역할별 페이지 권한 할당
3. **R3 페이지 기능 매핑** - 동작을 페이지 기능에 매핑
4. **R4 페이지 흐름 설계** - 탐색 및 페이지 흐름 설계
5. **R5 페이지 구조** - 데이터 특성에 따라 레이아웃 추천
6. **R6 페이지 인터랙션** - 인터랙션 패턴 결정
7. **R7 코드 생성** - HTML/CSS/JS 생성 (프로토타이퍼에 위임)
8. **R8 프로세스 검증** - 비즈니스 프로세스로 엔드투엔드 검증
9. **R9 인도** - 프로토타입 인도, 체험 안내

**산출물**: `prototype/` (프로세스 주도 워크벤치 및 페이지 체험 팁이 포함된 인터랙티브 프로토타입)

### 프로세스 관리

- **이어서 진행**: 중단 후 마지막 체크포인트부터 계속
- **설계 반복**: 설계 중 및 프로토타입 인도 후 변경 지원, 자동 영향도 평가
- **변경 수준**: L1 사소한 조정 → L4 주요 변경, 각각 다른 롤백 범위 적용

## 핵심 개념 (비기술 언어)

| FBS 계층 | 기술 용어 | 우리의 표현 | 예시 |
|-----------|---------------|----------|---------|
| F 기능 | Actor | 사용자 역할 | 관리자, 구매 담당자, 고객 |
| F 기능 | Scenario | 사용 사례 | 일일 체크인, 월간 재고 조사 |
| F 기능 | Process | 비즈니스 프로세스 | 주문부터 배송까지의 전체 프로세스 |
| S 구조 | Business Object | 관리할 대상 | 고객, 주문, 제품 |
| S 구조 | Attribute | 정보/필드 | 이름, 전화번호, 수량 |
| S 구조 | Relationship | 연관/연결 | 고객은 여러 주문을 "가짐" |
| B 동작 | State | 현재 상태 | 결제 대기, 배송 완료 |
| B 동작 | Operation | 가능한 동작 | 승인, 배송, 취소 |
| B 동작 | Business Rule | 제약 조건 | 관리자만 승인 가능 |

## 설계 원칙

1. **사용자 우선** - 사용자의 요구와 이해를 기술적 완벽함보다 우선
2. **점진적 공개** - 단계적으로 깊이를 더하고, 정보 과부하 방지
3. **즉각적 피드백** - 각 단계 후 즉시 결과 표시
4. **투명하고 통제 가능** - 사용자가 항상 대화 방향을 통제
5. **추적 가능** - 모든 결정과 변경 기록

## 품질 기준

### 완전성
- ✅ 모든 필수 필드가 작성됨
- ✅ 엔티티 속성 ≥ 2
- ✅ 관계가 명확히 정의됨
- ✅ 핵심 동작에 비즈니스 규칙 존재

### 일관성
- ✅ 통일된 명명 규칙
- ✅ 순환 종속성 없음
- ✅ 충돌하는 비즈니스 규칙 없음

### 신뢰도
- 90%-100%: 우수 - 다음 단계로 진행
- 70%-90%: 양호 - 진행 여부 확인
- 50%-70%: 보통 - 추가 명확화 권장
- <50%: 낮음 - 핵심 문제 반드시 해결

## 변경 관리

설계 중 및 프로토타입 인도 후 모두 변경을 지원합니다. 자세한 내용은 `references/iteration-rules.md`를 참조하세요.

4단계 변경을 지원합니다:

| 단계 | 유형 | 예시 | 처리 방법 |
|-------|------|---------|----------|
| L1 | 사소한 조정 | 표시 이름 변경 | 현재 단계에서 직접 수정 |
| L2 | 부분 수정 | 속성 유형 변경 | 관련 단계로 되돌아가서 수정 |
| L3 | 중간 변경 | 새 엔티티 추가 | 단계 시작점으로 되돌아감 |
| L4 | 주요 변경 | 엔티티 삭제 | 예비 설계로 되돌아가거나 새 프로젝트 생성 |

**변경 전**: 영향도 분석 표시
**변경 후**: 일관성 검증, 모델과 프로토타입 동기화

2. **시작** - `/mims design` 명령어 실행
3. **예비 설계** - 가이드의 질문에 답변 (P1-P6)
4. **상세 설계** - 객체, 상태, 규칙을 심층 분석 (D1-D5)
5. **프로토타입 생성** - 모델에서 프로토타입 생성 (R1-R9)
6. **반복** - 피드백에 따라 설계 조정 (`/mims change`)
7. **검증** - 브라우저에서 프로토타입 확인

## 설계 철학

MIMS는 다음 방법론과 개념에서 영감을 받았습니다:

- **FBS 온톨로지 (Gero, 1990)** - Function-Behavior-Structure, 모델링 프레임워크의 이론적 기반
- **UML (통합 모델링 언어)** - 유스케이스 다이어그램(F 계층), 클래스 다이어그램(S 계층), 상태 다이어그램(B 계층)
- **도메인 주도 설계 (DDD)** - 도메인 객체, 애그리거트, 도메인 이벤트
- **하니스 엔지니어링 (2026)** - 품질 주도 에이전트 제약 시스템
- **사용자 스토리 매핑** - 시나리오 주도 요구사항 분석
- **대화형 설계** - 자연어 인터랙션 설계

참고: `docs/core/DESIGN.md`
