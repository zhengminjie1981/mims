# MIMS - Make Idea Make Sense

> **아이디어를 합리적이고, 명확하고, 실현 가능하게**

---

## 프로젝트 개요

MIMS는 "미오사(MIMS Guide)"라는 AI 에이전트로, 비기술 사용자를 대화를 통해 소프트웨어 설계로 안내하고, 모호한 아이디어를 명확한 도메인 모델과 인터랙티브 프로토타입으로 변환합니다.

MIMS는 Claude Code CLI 사용자를 위해 배포되며, 메인 에이전트에 주입되는 페르소나 설정(`CLAUDE.md`), 구조화된 워크플로(Skill), 그리고 원자 작업을 처리하는 서브 에이전트의 세 가지 핵심 구성 요소로 이루어져 있습니다.

### 핵심 개념

**Make Idea Make Sense** - 구조화된 대화와 AI 지원을 통해 아이디어에서 프로토타입으로의 변환을 돕습니다:

- **아이디어를 명확하게** - 모호한 요구사항을 체계적이고 이해 가능한 개념으로 변환
- **아이디어를 가시적으로** - 추상적인 개념을 시각화하고 구체화
- **아이디어를 실행 가능하게** - 아이디어를 인터랙티브 프로토타입으로 변환

### 대상 사용자

- **제품 관리자** - 제품 아이디어를 빠르게 검증
- **비즈니스 애널리스트** - 비즈니스 요구사항을 명확하게 표현
- **창업가** - MVP(Minimum Viable Product) 계획
- **도메인 전문가** - 기술적 배경 없이 소프트웨어 설계

### 주요 특징

1. **기술적 장벽 없음** - 일상 언어로 대화, 도구 학습 불필요
2. **AI 주도 가이던스** - 에이전트가 적극적으로 질문하여 요구사항 명확화
3. **자동 모델 생성** - 대화에서 도메인 객체 모델 자동 구축
4. **즉각적 시각화** - 모델 변화와 관계를 실시간 표시
5. **인터랙티브 프로토타입** - 브라우저에서 직접 실행 가능한 프로토타입 생성

## 프로젝트 이름 의미

**MIMS** = **M**ake **I**dea **M**ake **S**ense

**발음**: /mɪmz/ (밈즈)

**전체 이름**: Make Idea Make Sense

- **Make** - 만들다, 구축하다, ...하게 하다
- **Idea** - 아이디어, 개념, 요구사항, 영감
- **Make Sense** - 합리적, 명확, 체계적, 이해 가능해지다

## 작동 원리

```
┌─────────────────────────────────────────────────────────────┐
│                     MIMS 대화 루프                          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  사용자가 아이디어/요구사항 제시                              │
│         ↓                                                   │
│  AI 에이전트가 질문 (명확화, 세부화, 보완)                     │
│         ↓                                                   │
│  사용자 응답                                                 │
│         ↓                                                   │
│  에이전트가 도메인 모델 자동 업데이트                         │
│         ↓                                                   │
│  모델을 사용자에게 표시하여 확인                              │
│         ↓                                                   │
│  사용자 피드백/확인                                          │
│         ↓                                                   │
│  모델이 명확하고 완전해질 때까지 루프 계속                    │
│         ↓                                                   │
│  인터랙티브 프로토타입 생성                                   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**모델링 프레임워크**: MIMS는 **FBS(Function-Behavior-Structure)** 프레임워크를 채택합니다. 이는 디자인 사이언스 분야의 고전적 존재론(Gero, 1990)이며, UML의 세 가지 핵심 다이어그램(유스케이스/상태/클래스 다이어그램)과도 높은 대응을 보입니다.

| FBS 계층 | 핵심 질문 | 대화 단계 |
|---------|----------|----------|
| **F 기능 계층** | 누가 사용? 어떤 시나리오? 무엇을? | 역할 및 시나리오 분석 |
| **B 동작 계층** | 어떤 상태? 어떻게 조작? 어떤 규칙? | 상태 및 조작 모델링 |
| **S 구조 계층** | 무엇을 관리? 어떤 정보? 어떻게 연관? | 비즈니스 객체 및 관계 모델링 |

> 대화 순서: 하향식(F→B→S), 모델 출력: 의존 순서(S→B→F)

## 빠른 시작

### 원라인 설치

**모든 사용자를 위한 추천 방식**:

```bash
# Linux / macOS
cd /your-project
curl -sSL https://raw.githubusercontent.com/zhengminjie1981/mims/main/install/install.sh | bash
```

```powershell
# Windows PowerShell
cd C:\your-project
iex ((New-Object System.Net.WebClient).DownloadString('https://raw.githubusercontent.com/zhengminjie1981/mims/main/install/install.ps1'))
```

**설치 과정**:
1. 스크립트 실행 후 설치 방법 선택:
   - **방식 1: 자동 다운로드 설치 (추천)** - GitHub에서 자동 다운로드 및 설치
   - **방식 2: 수동 다운로드 설치** - GitLab/GitHub에서 수동으로 zip 다운로드 후 경로 입력
2. 스크립트는 **현재 디렉토리**에 파일 설치
3. 완료 후 바로 사용 가능

**기업 인트라넷 사용자**: GitHub에 접근할 수 없는 경우, **방식 2**를 사용하여 GitLab에서 수동으로 zip 파일을 다운로드하세요.

상세 설치 가이드: [install/README.md](install/README.md)

## 두 단계

### 1단계: 요구사항 모델링

대화를 통해 완료 (9단계):
1. **요구사항 수집** - 초기 아이디어 이해
2. **컨텍스트 이해** - 비즈니스 배경과 사용자 역할 이해
3. **역할 및 시나리오 분석** - 사용 시나리오와 참여자 정의
4. **비즈니스 프로세스 매핑** - 핵심 비즈니스 프로세스 매핑
5. **비즈니스 객체 식별** - 관리할 대상 추출
6. **객체 관계 모델링** - 객체 간 관계와 모듈 구성 정의
7. **상태 및 전이 모델링** - 객체 상태와 전이 조건 정의
8. **조작 및 규칙 모델링** - 조작과 비즈니스 규칙 정의
9. **모델 검증** - 모델 완전성 확인

**출력**: `domain-model.yaml` (도메인 객체 모델)

### 2단계: 프로토타입 생성

모델에서 생성:
1. **인터페이스 매핑** - 페이지 구조 설계
2. **인터랙션 스타일** - 표시 형식 결정 (테이블/카드/리스트)
3. **코드 생성** - HTML/CSS/JS 생성
4. **미리보기 및 테스트** - 브라우저에서 프로토타입 보기
5. **반복 및 최적화** - 피드백에 따라 조정

**출력**: `prototype/` (인터랙티브 프로토타입)

## 핵심 개념 (비기술 언어)

| FBS 계층 | 기술 용어 | 우리의 표현 | 예시 |
|---------|----------|-----------|-----|
| F 기능 계층 | 액터 | 사용자 역할 | 관리자, 구매 담당자, 고객 |
| F 기능 계층 | 시나리오 | 사용 시나리오 | 일일 입고, 월간 재고 조사 |
| F 기능 계층 | 프로세스 | 비즈니스 프로세스 | 주문에서 배송까지의 전체 프로세스 |
| S 구조 계층 | 비즈니스 객체 | 관리할 것 | 고객, 주문, 제품 |
| S 구조 계층 | 속성 | 정보/필드 | 이름, 전화, 수량 |
| S 구조 계층 | 관계 | 연관/연결 | 고객이 여러 주문을 "가짐" |
| B 동작 계층 | 상태 | 현재 상황 | 결제 대기, 배송 완료 |
| B 동작 계층 | 조작 | 할 수 있는 것 | 승인, 배송, 취소 |
| B 동작 계층 | 비즈니스 규칙 | 제약 조건 | 관리자만 승인 가능 |

## 설계 원칙

1. **사용자 우선** - 사용자의 요구와 이해를 기술적 완벽성보다 우선
2. **점진적 공개** - 점차적으로 심화, 정보 과부하 방지
3. **즉각적 피드백** - 각 단계 후 즉시 결과 표시
4. **투명하고 제어 가능** - 사용자가 항상 대화 방향 제어
5. **추적 가능** - 모든 결정과 변경 기록

## 품질 기준

### 완전성
- ✅ 모든 필수 필드 입력됨
- ✅ 엔티티 속성 ≥2
- ✅ 명확한 관계 정의
- ✅ 핵심 조작에 비즈니스 규칙 있음

### 일관성
- ✅ 통일된 명명 규칙
- ✅ 순환 의존성 없음
- ✅ 충돌하는 비즈니스 규칙 없음

### 신뢰도
- 90%-100%: 우수 - 다음 단계 진행 권장
- 70%-90%: 양호 - 계속할지 질문
- 50%-70%: 보통 - 추가 명확화 권장
- <50%: 낮음 - 핵심 문제 해결 필요

## 변경 관리

4단계 변경 지원:

| 수준 | 유형 | 예시 | 처리 방법 |
|-----|------|-----|----------|
| L1 | 경미한 조정 | 중국어 이름 변경 | 현재 단계에서 직접 변경 |
| L2 | 국부적 수정 | 속성 유형 변경 | 관련 단계로 돌아감 |
| L3 | 중간 정도 변경 | 새 엔티티 추가 | 단계 시작점으로 돌아감 |
| L4 | 중대한 변경 | 엔티티 삭제 | 1단계로 돌아가거나 새 프로젝트 생성 |

**변경 전**: 영향 분석 표시
**변경 후**: 일관성 검증

## 설계 사상 출처

이 프로젝트는 다음 방법과 개념에서 착안했습니다:

- **FBS 존재론 (Gero, 1990)** - Function-Behavior-Structure, 모델링 프레임워크의 이론적 기반
- **UML (통합 모델링 언어)** - 유스케이스 다이어그램 (F 계층), 클래스 다이어그램 (S 계층), 상태 다이어그램 (B 계층)
- **도메인 주도 설계 (DDD)** - 도메인 객체, 애그리거트, 도메인 이벤트
- **Harness Engineering (2026)** - 품질 기반 에이전트 제약 시스템
- **사용자 스토리 매핑** - 시나리오 기반 요구사항 분석
- **대화형 디자인** - 자연어 인터랙션 디자인

상세: `docs/core/DESIGN.md`
