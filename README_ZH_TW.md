# MIMS - Make Idea Make Sense

> **讓想法變得合理、清晰、可落地**

---

## 專案簡介

MIMS 是一個 AI 智能體（Agent），化身「迷悟師」，通過主動式多輪對話引導非技術人員完成軟體設計，幫助使用者將模糊的想法轉化為清晰的領域模型和可互動的原型。

MIMS 面向 Claude Code CLI 與 Codex App 使用者部署，核心由三部分構成：注入主 Agent 的人設配置（`CLAUDE.md`）、結構化工作流（Skill）、以及處理原子任務的子代理（Sub-agents）。

### 核心理念

**Make Idea Make Sense** - 通過結構化的對話和 AI 輔助，幫助使用者完成從想法到原型的轉化：

- **讓想法清晰** - 將模糊的需求變得有條理、可理解
- **讓想法可見** - 將抽象的概念可視化、具象化
- **讓想法可落地** - 將想法轉化為可互動的原型

### 目標使用者

- 產品經理 - 快速驗證產品想法
- 業務分析師 - 清晰表達業務需求
- 創業者 - 規劃 MVP（最小可行產品）
- 領域專家 - 無需技術背景也能設計軟體

### 核心特點

1. **零技術門檻** - 用日常語言對話，無需學習任何工具
2. **AI 主動引導** - Agent 主動提問，幫助澄清需求
3. **自動生成模型** - 基於對話自動構建領域物件模型
4. **即時視覺化** - 即時展示模型變化和關係圖
5. **可互動原型** - 生成可直接在瀏覽器執行的原型

## 工作原理

```
┌─────────────────────────────────────────────────────────────┐
│                     MIMS 對話循環                             │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  使用者提出想法/需求                                          │
│         ↓                                                   │
│  AI Agent 主動提問（澄清、細化、補全）                         │
│         ↓                                                   │
│  使用者回答                                                  │
│         ↓                                                   │
│  Agent 自動更新領域模型                                       │
│         ↓                                                   │
│  展示模型給使用者確認                                          │
│         ↓                                                   │
│  使用者回饋/確認                                              │
│         ↓                                                   │
│  循環繼續，直到模型清晰完整                                    │
│         ↓                                                   │
│  生成可互動原型                                               │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**建模框架**：MIMS 採用 **FBS（Function-Behavior-Structure）** 框架，這是設計科學領域的經典本體論（Gero, 1990），也與 UML 三類核心圖（用例圖/狀態圖/類圖）高度對應。

| FBS 層 | 核心問題 | 對話階段 |
|--------|---------|---------|
| **F 功能層** | 誰在用？什麼場景？做什麼事？ | 角色與場景分析 |
| **B 行為層** | 有哪些狀態？怎麼操作？有什麼規則？ | 狀態與操作建模 |
| **S 結構層** | 管理什麼？有哪些資訊？如何關聯？ | 業務物件與關係建模 |

> 對話順序自頂向下（F→B→S），模型輸出按依賴順序（S→B→F）。


### 技術選型

| 層次 | 技術 | 理由 |

| 人設與基礎行為 | CLAUDE.md / AGENTS.md | T0 注入主 Agent，始終生效 |

| 原型生成 | HTML/CSS/JS | 零依賴、瀏覽器直接執行 |
| 文件輸出 | Markdown | 易讀、易版本管理 |

## 三個階段

### 階段一：初步設計

通過對話完成（6 個步驟）：
1. **P1 需求收集與準備** - 了解初步想法，引導資料準備
2. **P2 資料理解**（可選）- 分析使用者提供的資料
3. **P3 上下文與目標** - 理解業務背景和整體目標
4. **P4 角色與場景** - 定義使用者角色和使用場景（多對多）
5. **P5 業務流程** - 梳理核心流程（場景掛載，資訊流）
6. **P6 架構概覽** - 模組劃分、外部介面、AI Agent 評估

**檢查點**：初步驗證 → 生成 `srs.md`

**輸出**：`domain-model.yaml`（F 層 + 模組）+ `srs.md`（軟體需求規格說明書）

### 階段二：詳細設計

深入建模（5 個步驟）：
1. **D1 業務物件識別** - 提取要管理的事物及其屬性
2. **D2 關係與模組分配** - 定義關係、模組分配、AI Agent 設計
3. **D3 狀態與生命週期** - 定義物件狀態和轉換條件
4. **D4 操作與業務規則** - 定義操作和業務規則
5. **D5 模型驗證與置信度** - 驗證模型完整性，評估置信度

**檢查點**：完整驗證 → 生成 `sdd.md`

**輸出**：`domain-model.yaml`（完整 FBS）+ `sdd.md`（軟體設計文件）

### 階段三：原型生成

基於模型生成（9 個步驟）：
1. **R1 模型分析** - 分析模型資料特徵
2. **R2 頁面權限** - 按角色分配頁面權限
3. **R3 頁面功能映射** - 將操作映射到頁面功能
4. **R4 頁面流程設計** - 設計導航和頁面流程
5. **R5 頁面結構** - 根據資料特徵推薦版面配置
6. **R6 頁面互動** - 確定互動模式
7. **R7 程式碼生成** - 生成 HTML/CSS/JS（委託給原型生成子代理）
8. **R8 流程驗證** - 用業務流程端到端驗證原型
9. **R9 交付** - 交付原型，引導使用者體驗

**輸出**：`prototype/`（可互動原型，含流程驅動的工作台和頁面體驗提示）

### 流程管理

- **恢復**：中斷後從上次檢查點繼續
- **設計迭代**：支援設計中和原型交付後的變更，自動影響評估
- **變更等級**：L1 輕微調整 → L4 重大變更，各有不同的回退範圍

## 核心概念（非技術語言）

| FBS 層 | 專業術語 | 我們的說法 | 舉例 |
|--------|---------|-----------|------|
| F 功能層 | 參與者（Actor） | 使用者角色 | 管理員、採購員、客戶 |
| F 功能層 | 場景（Scenario） | 使用場景 | 日常入庫、月度盤點 |
| F 功能層 | 流程（Process） | 業務流程 | 從下單到收貨的全流程 |
| S 結構層 | 業務物件（Object） | 要管理的東西 | 客戶、訂單、產品 |
| S 結構層 | 屬性（Attribute） | 資訊/特徵/欄位 | 姓名、電話、數量 |
| S 結構層 | 關係（Relationship） | 關聯/連接 | 客戶「擁有」多個訂單 |
| B 行為層 | 狀態（State） | 當前狀況 | 待支付、已發貨 |
| B 行為層 | 操作（Operation） | 可以做的事 | 審批、發貨、取消 |
| B 行為層 | 業務規則（Rule） | 約束條件 | 只有管理員可以審批 |

## 設計原則

1. **使用者優先** - 使用者的需求和理解優先於技術完美
2. **漸進披露** - 逐步深入，避免資訊過載
3. **即時回饋** - 每步操作後立即展示結果
4. **透明可控** - 使用者始終掌控對話方向
5. **可追溯** - 記錄所有決策和變更

## 品質標準

### 完整性
- ✅ 所有必需欄位已填寫
- ✅ 實體屬性≥2 個
- ✅ 關係定義清晰
- ✅ 核心操作有業務規則

### 一致性
- ✅ 命名規範統一
- ✅ 無循環依賴
- ✅ 業務規則無衝突

### 置信度
- 90%-100%：優秀 - 建議進入下一步
- 70%-90%：良好 - 詢問是否繼續
- 50%-70%：中等 - 建議繼續澄清
- <50%：較低 - 必須解決關鍵問題

## 變更管理

設計中和原型交付後均可進行變更。詳見 `references/iteration-rules.md`。

支援 4 級變更：

| 級別 | 類型 | 範例 | 處理方式 |
|------|------|------|---------|
| L1 | 輕微調整 | 修改顯示名稱 | 當前步驟直接修改 |
| L2 | 局部修改 | 修改屬性類型 | 回退到相關步驟 |
| L3 | 中等變更 | 新增實體 | 回退到階段起點 |
| L4 | 重大變更 | 刪除實體 | 回退到初步設計或建立新專案 |

**變更前**：展示影響分析
**變更後**：驗證一致性，同步模型和原型

## 設計思想來源

本專案借鑑了以下方法和理念：

- **FBS 本體論（Gero, 1990）** - Function-Behavior-Structure，建模框架的理論基礎
- **UML（統一建模語言）** - 用例圖（F 層）、類圖（S 層）、狀態圖（B 層）
- **領域驅動設計（DDD）** - 領域物件、聚合、領域事件
- **Harness Engineering（2026）** - 品質驅動的 Agent 約束系統
- **使用者故事地圖** - 場景驅動的需求分析
- **對話式設計** - 自然語言互動設計

詳見：`docs/core/DESIGN.md`
