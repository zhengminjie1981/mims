# MIMS - Make Idea Make Sense

> **让想法变得合理、清晰、可落地**

---

## 项目简介

MIMS 是一个 AI 智能体（Agent），化身"迷悟师"，通过主动式多轮对话引导非技术人员完成软件设计，帮助用户将模糊的想法转化为清晰的领域模型和可交互的原型。

MIMS 面向 Claude Code CLI 用户部署，核心由三部分构成：注入主 Agent 的人设配置（`CLAUDE.md`）、结构化工作流（Skill）、以及处理原子任务的子代理（Sub-agents）。

### 核心理念

**Make Idea Make Sense** - 通过结构化的对话和AI辅助，帮助用户完成从想法到原型的转化：

- **让想法清晰** - 将模糊的需求变得有条理、可理解
- **让想法可见** - 将抽象的概念可视化、具象化
- **让想法可落地** - 将想法转化为可交互的原型

### 目标用户

- 产品经理 - 快速验证产品想法
- 业务分析师 - 清晰表达业务需求
- 创业者 - 规划MVP（最小可行产品）
- 领域专家 - 无需技术背景也能设计软件

### 核心特点

1. **零技术门槛** - 用日常语言对话，无需学习任何工具
2. **AI主动引导** - Agent主动提问，帮助澄清需求
3. **自动生成模型** - 基于对话自动构建领域对象模型
4. **即时可视化** - 实时展示模型变化和关系图
5. **可交互原型** - 生成可直接在浏览器运行的原型

## 工作原理

```
┌─────────────────────────────────────────────────────────────┐
│                     MIMS 对话循环                             │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  用户提出想法/需求                                            │
│         ↓                                                   │
│  AI Agent主动提问（澄清、细化、补全）                          │
│         ↓                                                   │
│  用户回答                                                    │
│         ↓                                                   │
│  Agent自动更新领域模型                                       │
│         ↓                                                   │
│  展示模型给用户确认                                          │
│         ↓                                                   │
│  用户反馈/确认                                               │
│         ↓                                                   │
│  循环继续，直到模型清晰完整                                    │
│         ↓                                                   │
│  生成可交互原型                                               │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**建模框架**：MIMS 采用 **FBS（Function-Behavior-Structure）** 框架，这是设计科学领域的经典本体论（Gero, 1990），也与 UML 三类核心图（用例图/状态图/类图）高度对应。

| FBS 层 | 核心问题 | 对话阶段 |
|--------|---------|---------|
| **F 功能层** | 谁在用？什么场景？做什么事？ | 角色与场景分析 |
| **B 行为层** | 有哪些状态？怎么操作？有什么规则？ | 状态与操作建模 |
| **S 结构层** | 管理什么？有哪些信息？如何关联？ | 业务对象与关系建模 |

> 对话顺序自顶向下（F→B→S），模型输出按依赖顺序（S→B→F）。

## 三个阶段

### 阶段一：初步设计（Preliminary Design）

通过对话完成（6 个步骤）：
1. **P1 需求收集与准备** - 了解初步想法，引导用户准备资料
2. **P2 资料理解**（可选）- 理解用户提供的材料
3. **P3 背景与目标** - 理解业务背景和总体目标
4. **P4 角色与场景** - 定义用户角色和使用场景（多对多关联）
5. **P5 业务流程** - 梳理核心业务流程（场景挂载、信息传递）
6. **P6 总体架构** - 模块划分、外部接口、AI Agent 评估

**检查点**：初步验证 → 生成 `srs.md`

**输出**：`domain-model.yaml`（F 层 + 模块）+ `srs.md`（软件需求规格说明书）

### 阶段二：详细设计（Detailed Design）

深入建模（5 个步骤）：
1. **D1 业务对象识别** - 提取要管理的事物及其属性
2. **D2 对象关系与模块归属** - 定义对象间关系、模块归属、AI Agent 设计
3. **D3 状态与生命周期** - 定义对象状态和转换条件
4. **D4 操作与业务规则** - 定义操作和业务规则
5. **D5 模型验证与置信度** - 验证模型完整性，评估置信度

**检查点**：完整验证 → 生成 `sdd.md`

**输出**：`domain-model.yaml`（完整 FBS）+ `sdd.md`（软件设计规格说明书）

### 阶段三：原型生成（Prototype）

基于模型生成（9 个步骤）：
1. **R1 建模结果分析** - 分析模型数据特点
2. **R2 页面权限规划** - 根据角色分配页面权限
3. **R3 页面功能映射** - 将操作映射到页面功能
4. **R4 页面流程设计** - 设计页面间导航和流程
5. **R5 页面结构设计** - 根据数据特点推荐布局
6. **R6 页面交互设计** - 确定交互方式
7. **R7 代码生成** - 生成 HTML/CSS/JS（委托 prototyper）
8. **R8 流程验证** - 用业务流程端到端验证
9. **R9 交付** - 交付原型，引导用户体验

**输出**：`prototype/`（可交互原型，含流程驱动的工作台和页面体验提示）

### 过程管理

- **中断恢复**：对话中断后可从上次进度继续
- **设计迭代**：支持设计过程中和原型交付后的变更，自动评估影响并同步更新模型和原型
- **变更级别**：L1 轻微调整 → L4 重大变更，不同级别对应不同回退范围

## 核心概念（非技术语言）

| FBS 层 | 专业术语 | 我们的说法 | 举例 |
|--------|---------|-----------|------|
| F 功能层 | 参与者（Actor） | 用户角色 | 管理员、采购员、客户 |
| F 功能层 | 场景（Scenario） | 使用场景 | 日常入库、月度盘点 |
| F 功能层 | 流程（Process） | 业务流程 | 从下单到收货的全流程 |
| S 结构层 | 业务对象（Object） | 要管理的东西 | 客户、订单、产品 |
| S 结构层 | 属性（Attribute） | 信息/特征/字段 | 姓名、电话、数量 |
| S 结构层 | 关系（Relationship） | 关联/连接 | 客户"拥有"多个订单 |
| B 行为层 | 状态（State） | 当前状况 | 待支付、已发货 |
| B 行为层 | 操作（Operation） | 可以做的事 | 审批、发货、取消 |
| B 行为层 | 业务规则（Rule） | 约束条件 | 只有管理员可以审批 |

## 设计原则

1. **用户优先** - 用户的需求和理解优先于技术完美
2. **渐进披露** - 逐步深入，避免信息过载
3. **即时反馈** - 每步操作后立即展示结果
4. **透明可控** - 用户始终掌控对话方向
5. **可追溯** - 记录所有决策和变更

## 质量标准

### 完整性
- ✅ 所有必需字段已填写
- ✅ 实体属性≥2个
- ✅ 关系定义清晰
- ✅ 核心操作有业务规则

### 一致性
- ✅ 命名规范统一
- ✅ 无循环依赖
- ✅ 业务规则无冲突

### 置信度
- 90%-100%：优秀 - 建议进入下一步
- 70%-90%：良好 - 询问是否继续
- 50%-70%：中等 - 建议继续澄清
- <50%：较低 - 必须解决关键问题

## 变更管理

设计过程中和原型交付后均支持变更。详情见 `references/iteration-rules.md`。

支持 4 级变更：

| 级别 | 类型 | 示例 | 处理方式 |
|------|------|------|---------|
| L1 | 轻微调整 | 修改中文名称 | 当前步骤直接修改 |
| L2 | 局部修改 | 修改属性类型 | 回退到相关步骤 |
| L3 | 中等变更 | 新增实体 | 回退到阶段起点 |
| L4 | 重大变更 | 删除实体 | 回退到初步设计或创建新项目 |

**变更前**：展示影响分析
**变更后**：验证一致性，同步更新模型和原型

### 作为用户

1. **准备** - 有一个初步的想法或需求
2. **启动** - 调用 `/mims design` 命令
3. **初步设计** - 回答迷悟师的引导问题（P1-P6）
4. **详细设计** - 深入定义对象、状态、规则（D1-D5）
5. **生成原型** - 基于模型生成原型（R1-R9）
6. **迭代** - 根据反馈调整设计（`/mims change`）
7. **验证** - 在浏览器中查看原型

## 设计思想来源

本项目借鉴了以下方法和理念：

- **FBS 本体论（Gero, 1990）** - Function-Behavior-Structure，建模框架的理论基础
- **UML（统一建模语言）** - 用例图（F层）、类图（S层）、状态图（B层）
- **领域驱动设计（DDD）** - 领域对象、聚合、领域事件
- **Harness Engineering（2026）** - 质量驱动的Agent约束系统
- **用户故事地图** - 场景驱动的需求分析
- **对话式设计** - 自然语言交互设计

详见：`docs/core/DESIGN.md`
